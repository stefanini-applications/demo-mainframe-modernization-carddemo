1. Overview & Purpose

The transaction-types module is an optional extension to the CardDemo mainframe credit card management system. It provides administrative management of transaction type reference data stored in IBM DB2 relational tables rather than VSAM files.

Transaction type codes (e.g., 01=PURCHASE, 02=PAYMENT) are recorded on every transaction record in the TRANFILE VSAM file via the TRAN-TYPE-CD field. This module allows administrators to maintain the lookup table that classifies and describes those codes, as well as nested category sub-codes that drive interest rate groupings for billing calculations.

Optional Module: Requires IBM DB2 to be provisioned and configured in the z/OS environment. The core CardDemo application (VSAM/CICS) functions independently; this module extends it with DB2-backed reference data management.

Location in repository: app/app-transaction-type-db2/

Primary responsibilities:

  • Maintain CARDDEMO.TRANSACTION_TYPE DB2 table (2-char type codes and descriptions)
  • Maintain CARDDEMO.TRANSACTION_TYPE_CATEGORY DB2 table (sub-categories per type with FK constraint)
  • Provide CICS online screens for admin-only add, edit, delete, and list operations
  • Provide batch job (MNTTRDB2) for bulk insert/update/delete from flat-file input
  • Extract transaction type data to flat files (TRANEXTR) for downstream TRANREPT reporting

2. Architectural Context

This module sits at the boundary between the traditional VSAM-based CardDemo application and an optional DB2 relational database layer. It follows the same CICS/BMS pattern as other CardDemo modules but replaces VSAM file I/O with embedded SQL (EXEC SQL) calls to DB2.

Communication between transactions uses the standard CARDDEMO-COMMAREA (COCOM01Y copybook), passing the user's identity and type. The admin-only access check is enforced on each screen entry using CDEMO-USER-TYPE.

The TRANEXTR job bridges the DB2 world back to VSAM-based batch processing by extracting DB2 type data to fixed-length flat files that the TRANREPT batch report program (CBTRN03C) can read directly.

3. Key Components

COTRTUPC.cbl — Transaction Type Update/Add CICS

Transaction: CTTU  |  Mapset: COTRTUP  |  Map: CTRTUPA

Handles both add and edit operations for a single transaction type record. Uses a multi-step confirm-before-commit pattern:

  • Step 1: Enter 2-char type code → DB2 SELECT to fetch existing record
  • Step 2: Display existing values (edit) or blank fields (add); user modifies description
  • Step 3: Validate input; show confirmation prompt ("Changes validated. Press F5 to save")
  • Step 4: PF5 commits — UPDATE if record existed, INSERT if new
  • PF4 triggers delete with 2-step confirmation flow
  • PF3 exits to Admin Menu (XCTL to COADM01C)
  • PF12 cancels and returns to search state

DB2 operations: SELECT, UPDATE, INSERT INTO, DELETE FROM CARDDEMO.TRANSACTION_TYPE. Issues EXEC CICS SYNCPOINT after each successful DML.

Key working-storage sections: TTYP-UPDATE-RECORD (type + description, 60 bytes), WS-NON-KEY-FLAGS (description validation), WS-DATACHANGED-FLAG (change detection), WS-INFO-MSG / WS-RETURN-MSG (user messages).

COTRTLIC.cbl — Transaction Type List/Browse CICS

Transaction: CTLI  |  Mapset: COTRTLI  |  Map: CTRTLIA

Provides a scrollable list (up to 7 rows per page) with optional filter by type code or description. Supports per-row action selection (U=Update, D=Delete).

  • Two DB2 cursors declared in working-storage: C-TR-TYPE-FORWARD (PF8 scroll) and C-TR-TYPE-BACKWARD (PF7 scroll)
  • PF7/PF8 for bi-directional pagination
  • PF2 navigates to COTRTUPC for add
  • PF10 confirms pending per-row U or D action
  • PF3 returns to Admin Menu
  • Enters row with U → PF10 → XCTL to COTRTUPC with type code
  • Uses CSDB2RWY.cpy (common DB2 working-storage) and CSDB2RPY.cpy (common DB2 procedures)

DB2 operations: Cursor SELECT with FETCH, DELETE FROM CARDDEMO.TRANSACTION_TYPE. Priming query on entry to verify DB2 availability.

Delete safety flags: WS-DELETE-STATUS, WS-UPDATE-STATUS, WS-ROW-SELECTION-CHANGED, WS-ACTIONS-SELECTED (counts deletes/updates/no-actions requested across 7 rows).

COBTUPDT.cbl — Batch Transaction Type Maintenance BATCH

JCL Job: MNTTRDB2  |  DB2 Plan: CARDDEMO

Batch program that reads sequential flat-file (INPFILE) and performs INSERT, UPDATE, or DELETE on CARDDEMO.TRANSACTION_TYPE. Input record layout (53 bytes):

  • Col 1: A=Add/Insert, U=Update, D=Delete, *=Comment (skip)
  • Cols 2–3: Transaction type code (2-char numeric)
  • Cols 4–53: Transaction type description (50 chars)

Abends (9999-ABEND) on any unrecognized action code or unexpected SQL error.

6. Processing Flows

CTTU — Add a New Transaction Type

1
Admin navigates from Admin Menu (CA00) → CTTU invoked → COTRTUPC starts; COMMAREA verified
2
Screen CTRTUPA displayed with empty TRTYPCD/TRTYDSC; message: "Enter transaction type to be maintained"
3
Admin types 2-char code; presses ENTER → DB2 SELECT executed. SQLCODE=100 (not found)
4
Message changes to "Enter new transaction type details." → admin enters description
5
Admin presses PF5 → input validated (alphanumeric, non-blank). Message: "Changes validated. Press F5 to save"
6
PF5 again → INSERT INTO DB2; EXEC CICS SYNCPOINT; message: "Changes committed to database"
7
Admin presses PF3 → XCTL to COADM01C (Admin Menu)

CTTU — Edit an Existing Transaction Type

1
Enter existing code → SELECT returns SQLCODE=0 → existing description displayed
2
Message: "Update transaction type details shown." Admin modifies description
3
1205-COMPARE-OLD-NEW compares TTUP-OLD-* vs TTUP-NEW-*. If identical: "No change detected..."
4
If changed and valid: "Changes validated. Press F5 to save." → PF5 → UPDATE; SYNCPOINT

CTTU — Delete a Transaction Type

1
Record displayed. Admin presses PF4 → message: "Delete this record? Press F4 to confirm"
2
Admin presses PF4 again → DELETE FROM executed; SYNCPOINT; "Delete successful."
3
FK constraint failure (type has categories): SQLCODE ≠ 0 → "Delete of record failed" displayed

CTLI — Browse and Filter Transaction Types

1
COTRTLIC starts → 9998-PRIMING-QUERY verifies DB2 connectivity (SELECT 1 FROM SYSIBM.SYSDUMMY1)
2
OPEN C-TR-TYPE-FORWARD cursor with optional WHERE on type/description filter fields
3
FETCH up to 7 rows → populate TRTSEL/TRTTYP/TRTDSC screen rows 1–7; PAGENO incremented
4
PF8 (forward): FETCH next page via C-TR-TYPE-FORWARD cursor
5
PF7 (backward): OPEN C-TR-TYPE-BACKWARD cursor, FETCH previous page
6
Enter 'U' in TRTSEL field → PF10 → XCTL to COTRTUPC with type code pre-filled
7
Enter 'D' in TRTSEL field → PF10 → DELETE FROM executed; page refreshed
8
PF2 navigates to COTRTUPC (blank) to add a new type; PF3 exits to Admin Menu

7. DB2 SQL Operations

COTRTUPC — Single-Record Operations

-- Fetch existing record
EXEC SQL
    SELECT TR_TYPE, TR_DESCRIPTION
      INTO :DCL-TR-TYPE,
           :DCL-TR-DESCRIPTION
      FROM CARDDEMO.TRANSACTION_TYPE
     WHERE TR_TYPE = :DCL-TR-TYPE
END-EXEC
-- SQLCODE 0   = found; display for edit
-- SQLCODE 100 = not found; prompt for new description (add flow)
-- Other       = error; call 9999-FORMAT-DB2-MESSAGE
-- Update existing record
EXEC SQL
    UPDATE CARDDEMO.TRANSACTION_TYPE
       SET TR_DESCRIPTION = :DCL-TR-DESCRIPTION
     WHERE TR_TYPE = :DCL-TR-TYPE
END-EXEC
EXEC CICS SYNCPOINT END-EXEC
-- Insert new record
EXEC SQL
    INSERT INTO CARDDEMO.TRANSACTION_TYPE
           (TR_TYPE, TR_DESCRIPTION)
    VALUES (:DCL-TR-TYPE,
            :DCL-TR-DESCRIPTION)
END-EXEC
EXEC CICS SYNCPOINT END-EXEC
-- Delete record
EXEC SQL
    DELETE FROM CARDDEMO.TRANSACTION_TYPE
     WHERE TR_TYPE = :DCL-TR-TYPE
END-EXEC
EXEC CICS SYNCPOINT END-EXEC

COTRTLIC — Cursor-Based Paging

-- Forward cursor (PF8 scroll, WORKING-STORAGE declaration)
EXEC SQL
    DECLARE C-TR-TYPE-FORWARD CURSOR FOR
        SELECT TR_TYPE
          FROM CARDDEMO.TRANSACTION_TYPE
         WHERE TR_TYPE >= :WS-FILTER-TYPE-START
           AND TR_DESCRIPTION LIKE :WS-FILTER-DESC-MASK
         ORDER BY TR_TYPE
END-EXEC

-- Backward cursor (PF7 scroll)
EXEC SQL
    DECLARE C-TR-TYPE-BACKWARD CURSOR FOR
        SELECT TR_TYPE
          FROM CARDDEMO.TRANSACTION_TYPE
         WHERE TR_TYPE <= :WS-FILTER-TYPE-END
           AND TR_DESCRIPTION LIKE :WS-FILTER-DESC-MASK
         ORDER BY TR_TYPE DESC
END-EXEC

-- DB2 connectivity priming query
EXEC SQL
    SELECT 1
      INTO :WS-DUMMY-DB2-INT
      FROM SYSIBM.SYSDUMMY1
     FETCH FIRST 1 ROW ONLY
END-EXEC

COBTUPDT — Batch SQL

-- Insert (action 'A')
EXEC SQL
    INSERT INTO CARDDEMO.TRANSACTION_TYPE
           (TR_TYPE, TR_DESCRIPTION)
    VALUES (:INPUT-REC-NUMBER, :INPUT-REC-DESC)
END-EXEC

-- Update (action 'U')
EXEC SQL
    UPDATE CARDDEMO.TRANSACTION_TYPE
       SET TR_DESCRIPTION = :INPUT-REC-DESC
     WHERE TR_TYPE = :INPUT-REC-NUMBER
END-EXEC

-- Delete (action 'D')
EXEC SQL
    DELETE FROM CARDDEMO.TRANSACTION_TYPE
     WHERE TR_TYPE = :INPUT-REC-NUMBER
END-EXEC

8. Batch Processing

MNTTRDB2 Job (COBTUPDT)

Bulk maintenance of TRANSACTION_TYPE from a flat input file. Useful for environment-specific seeding, bulk corrections, or scripted migrations without CICS online access.

Input File Format (INPFILE, FB, LRECL=53)

Col 1     : Action  A=Add  U=Update  D=Delete  *=Comment(skip)
Cols 2-3  : TR_TYPE  (2-char, e.g. '08')
Cols 4-53 : TR_DESCRIPTION  (50-char description)

Sample Input File

* Bulk update run 2026-03-26
A08BALANCE TRANSFER
U01PURCHASE - RETAIL
D06

Processing Logic

OPEN INPUT TR-RECORD
PERFORM UNTIL LASTREC = 'Y'
    READ TR-RECORD NEXT INTO WS-INPUT-REC
    EVALUATE INPUT-REC-TYPE
        WHEN 'A'    PERFORM 10031-INSERT-DB
        WHEN 'U'    PERFORM 10032-UPDATE-DB
        WHEN 'D'    PERFORM 10033-DELETE-DB
        WHEN '*'    CONTINUE
        WHEN OTHER  PERFORM 9999-ABEND
    END-EVALUATE
END-PERFORM
CLOSE TR-RECORD

TRANEXTR Job (Daily Extract)

Extracts DB2 transaction type and category data to FB/LRECL=60 flat files for use by TRANREPT. Runs daily; backs up prior version to GDG before overwriting.

StepActionOutput Dataset
STEP10Backup TRANTYPE.PS to GDG (+1)HLQ.TRANTYPE.BKUP(+1)
STEP20Backup TRANCATG.PS to GDG (+1)HLQ.TRANCATG.PS.BKUP(+1)
STEP30Delete previous run's work files—

TRANTYPE Extraction SQL

SELECT CAST(CONCAT(CONCAT(
   TR_TYPE,
   CAST(TR_DESCRIPTION AS CHAR(50))
   ),
   REPEAT('0',8)
) AS CHAR(60))
FROM CARDDEMO.TRANSACTION_TYPE
ORDER BY TR_TYPE;

TRANCATG Extraction SQL

SELECT CAST(
    TRC_TYPE_CODE
 || TRC_TYPE_CATEGORY
 || CAST(TRC_CAT_DATA AS CHAR(50))
 || REPEAT('0',4)
 AS CHAR(60))
FROM CARDDEMO.TRANSACTION_TYPE_CATEGORY
ORDER BY TRC_TYPE_CODE, TRC_TYPE_CATEGORY;

9. Business Rules & Validation

Access Control

  • CTTU and CTLI accessible only from Admin Menu (CA00)
  • CDEMO-USER-TYPE = 'A' required in COMMAREA; Regular Users (type 'U') cannot access
  • PF3 on any screen executes XCTL to COADM01C

Type Code Rules

  • Exactly 2 characters (CHAR(2) NOT NULL)
  • Unique — PRIMARY KEY enforced by DB2
  • Used as TRAN-TYPE-CD PIC X(02) in TRANFILE VSAM records
  • Deleting a type that has category children fails (FK ON DELETE RESTRICT)

Description Validation

  • VARCHAR(50) NOT NULL — blank description rejected
  • Validated with alphanumeric check (1230-EDIT-ALPHANUM-REQD paragraph in COTRTUPC)
  • Input error sets INPUT-ERROR flag; field highlighted with error attribute

Optimistic Change Detection (COTRTUPC)

  • Original fetched values stored in TTUP-OLD-* fields; user input in TTUP-NEW-* fields
  • 1205-COMPARE-OLD-NEW paragraph compares them before issuing UPDATE
  • No changes: "No change detected with respect to values fetched." — UPDATE skipped
  • External concurrent modification detected: "Record changed by some one else. Please review"

DB2 Error Handling

  • All EXEC SQL blocks followed by SQLCODE evaluation
  • SQLCODE=0: success; SQLCODE=100: not found; other: error
  • Errors route to 9999-FORMAT-DB2-MESSAGE (CSDB2RPY) → DSNTIAC utility formats SQLCA
  • Formatted message written to WS-RETURN-MSG for display on screen
  • Critical/unexpected errors: EXEC CICS ABEND via ABEND-ROUTINE paragraph

SYNCPOINT (Transaction Commit)

  • After every successful INSERT, UPDATE, or DELETE: EXEC CICS SYNCPOINT END-EXEC
  • PF12 (cancel) or PF3 (exit) without save: CICS task termination rolls back uncommitted DB2 UoW
  • Ensures atomicity: screen message and DB2 commit are tightly coupled

Batch MNTTRDB2 Rules

  • Lines beginning with * are comments and are skipped silently
  • Unrecognized action codes cause ABEND (9999-ABEND)
  • INSERT failures (e.g., duplicate key -803) cause ABEND
  • UPDATE/DELETE with SQLCODE ≠ 0 causes ABEND
  • No transactional rollback on partial failure — records processed before the failing record are committed

DB2 to Flat File Bridge (TRANEXTR)

TRANEXTR uses DSNTIAUL to serialize DB2 rows as fixed-length EBCDIC records. These become the TRANTYPE.PS and TRANCATG.PS datasets that TRANREPT (CBTRN03C) reads as sequential VSAM-substitute input when printing transaction type descriptions on the daily report — bridging DB2 data into the VSAM-centric batch world.

Flat File to DB2 Bridge (MNTTRDB2)

COBTUPDT reads a user-prepared sequential INPFILE and drives DB2 DML operations. Provides a scriptable bulk-maintenance channel without requiring CICS online terminal access. Suitable for automated environment provisioning, test data loading, or production data corrections via controlled JCL submission.

15. Development Guidelines & Patterns

Host Variable Handling

  • Always use DCLGEN-generated host-variable structures (DCLTRANSACTION-TYPE) rather than inline PIC definitions for DB2 columns
  • For VARCHAR fields, set both the length indicator and text field before every EXEC SQL DML:
MOVE INPUT-DESC TO DCL-TR-DESCRIPTION-TEXT
COMPUTE DCL-TR-DESCRIPTION-LEN =
    FUNCTION LENGTH(FUNCTION TRIM(DCL-TR-DESCRIPTION-TEXT))

SQLCODE Evaluation Pattern

EXEC SQL
    ... SQL statement ...
END-EXEC

EVALUATE TRUE
    WHEN SQLCODE = ZERO
        CONTINUE   -- success path
    WHEN SQLCODE = +100
        SET RECORD-NOT-FOUND TO TRUE
    WHEN OTHER
        MOVE SQLCODE TO WS-DISP-SQLCODE
        PERFORM 9999-FORMAT-DB2-MESSAGE
            THRU 9999-FORMAT-DB2-MESSAGE-EXIT
        SET WS-DB2-ERROR TO TRUE
END-EVALUATE

SYNCPOINT Pattern

-- After every successful DML:
EXEC CICS SYNCPOINT END-EXEC

-- Check response if needed:
IF WS-RESP-CD = DFHRESP(NORMAL)
    SET CONFIRM-UPDATE-SUCCESS TO TRUE
ELSE
    PERFORM ABEND-ROUTINE
END-IF

      INTO :DCL-TR-TYPE, :DCL-TR-DESCRIPTION
      FROM CARDDEMO.TRANSACTION_TYPE

* 3. SQLCODE=100 means new record -- INSERT

    EXEC SQL
        INSERT INTO CARDDEMO.TRANSACTION_TYPE

DB2 Bind/Compile Procedure

After any change to EXEC SQL statements, the full precompile-compile-link-bind cycle is required:

  1. DB2 COBOL precompiler — generates modified COBOL source + DBRM
  2. IBM Enterprise COBOL compile
  3. Link-edit into LOADLIB
  4. DBRM placed in DBRMLIB
  5. DB2 BIND PACKAGE and BIND PLAN for CARDDEMO

16. User Story Examples

Browse & View

  • As an administrator, I want to view all transaction type codes so I can understand the classification scheme used in transaction records
  • As an administrator, I want to filter the list by type code or description so I can quickly find a specific type without paging through all records

Add & Edit

  • As an administrator, I want to add a new transaction type code so new purchase categories can be tracked in the billing system
  • As an administrator, I want to correct the description of an existing type so reports and screens display accurate labels
  • As an administrator, I want to confirm changes before they are committed to DB2 so I can prevent accidental modifications
  • As a system, I want to detect when no changes were made before saving so unnecessary DB2 UPDATE operations are skipped

Delete

  • As an administrator, I want to delete an obsolete transaction type so the reference data stays current and reports are not cluttered with unused codes
  • As a system, I want to prevent deletion of a type that has associated category records so referential integrity is maintained in DB2 via the FK constraint

Batch Maintenance

  • As a system operator, I want to bulk-load type updates from a flat file so I can apply environment-specific tables without online screen entry
  • As a system, I want to extract transaction type data daily so TRANREPT batch reports contain current type descriptions
