🏷️ Transaction Types

User Story Development Guide — CardDemo Optional Module

Optional Module DB2 Admin Only CTTU · CTLI

📋 Module Overview

📦 Optional Module: Requires DB2 to be configured. Accessible from Admin Menu (CA00). Located in app/app-transaction-type-db2/.

Purpose: Maintain transaction type reference data in DB2 tables. Provides CICS-based CRUD for transaction types (add, update, list, delete) and batch synchronization via MNTTRDB2.

Business Context: Transaction types (TRAN-TYPE-CD in TRANFILE) classify financial transactions for reporting and interest calculation. This module uses DB2 to demonstrate relational database patterns including cursors, dynamic SQL, and batch DB2 operations — contrasting with the base application's VSAM approach.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • View transaction types: As an administrator, I want to view all transaction type codes and descriptions so I understand the transaction classification scheme
  • Add transaction type: As an administrator, I want to add a new transaction type code so new purchase categories can be tracked
  • Update transaction type: As an administrator, I want to update a transaction type description so the reference data is accurate
  • Delete transaction type: As an administrator, I want to delete an obsolete transaction type so the reference data stays clean
  • Sync batch: As a system operator, I want to run MNTTRDB2 so DB2 transaction type table stays synchronized

⚡ Development Acceleration Factors

  • DB2 cursor pattern: COTRTLIC demonstrates DB2 cursor for paginated browse — reuse pattern for any DB2 list function
    • DB2 INSERT/UPDATE pattern: COTRTUPC shows embedded SQL INSERT and UPDATE — template for new DB2 write operations
  • TRANEXTR extract pattern: DSNTIAUL unload pattern available for extracting any DB2 table to flat file
  • Batch DB2 maintenance: COBTUPDT and MNTTRDB2 JCL provide template for batch DB2 table updates

📊 Complexity Guidelines

Simple (1–2 pts) Add display column to type list; change DB2 column label on screen; update existing type description
Medium (3–5 pts) Add new DB2 column to transaction type table; implement type search by description; add type usage count query (join with TRANFILE data)
Complex (5–8 pts) Migrate TRANTYPE VSAM to DB2 (dual-mode read during transition); add DB2 stored procedure for type validation; implement type hierarchy (parent/child type codes); integrate type codes with reporting for dynamic category descriptions

🔧 Technical Foundation

DB2 Patterns Used

CTTU (COTRTUPC) — Add/Edit:
  EXEC SQL INSERT INTO TRAN_TYPE_TABLE
    (TYPE_CD, TYPE_DESC, ...)
    VALUES (:WS-TYPE-CD, :WS-TYPE-DESC, ...)
  END-EXEC

  EXEC SQL UPDATE TRAN_TYPE_TABLE
    SET TYPE_DESC = :WS-TYPE-DESC
    WHERE TYPE_CD = :WS-TYPE-CD
  END-EXEC

CTLI (COTRTLIC) — List with cursor:
  EXEC SQL DECLARE TRAN_TYPE_CURSOR CURSOR FOR
    SELECT TYPE_CD, TYPE_DESC FROM TRAN_TYPE_TABLE
    ORDER BY TYPE_CD
  END-EXEC

  EXEC SQL OPEN TRAN_TYPE_CURSOR END-EXEC
  EXEC SQL FETCH TRAN_TYPE_CURSOR
    INTO :WS-TYPE-CD, :WS-TYPE-DESC
  END-EXEC

  EXEC SQL DELETE FROM TRAN_TYPE_TABLE
    WHERE TYPE_CD = :WS-TYPE-CD
  END-EXEC

DB2 Setup Requirements

  • Run CREADB21 JCL to create database schema before first use
  • DB2 plan/package must be bound for COTRTUPC, COTRTLIC, COBTUPDT
  • DB2 subsystem must be accessible from CICS

🔗 Dependencies

  • authentication: Must be Admin user (CDEMO-USER-TYPE = 'A') — module only accessible from Admin Menu (CA00)
  • DB2 subsystem: Must be running and CardDemo database objects created via CREADB21
  • transactions: TRAN-TYPE-CD in TRANFILE must match codes defined in this module's DB2 table
  • DEFGDGD JCL: Additional GDG bases for DB2 data must be defined before TRANEXTR

✅ Acceptance Criteria Examples

List transaction types:
Given transaction type records exist in the DB2 table,
When admin navigates to CTLI,
Then all type codes and descriptions are displayed using DB2 cursor with PF7/PF8 pagination
Add transaction type:
Given a new unique type code entered on CTTU,
When pressing ENTER,
Then a DB2 INSERT is executed; the new type appears in the CTLI list; RC=0 on the SQL operation
Add duplicate type code:
Given a type code that already exists in the DB2 table,
When pressing ENTER on CTTU,
Then SQLCODE indicates duplicate key; error message displayed; no duplicate row inserted
Delete transaction type:
Given an admin selects a type code on CTLI and confirms deletion,
When pressing ENTER,
Then the DB2 DELETE executes; type no longer appears in list; existing TRAN-TYPE-CD references in TRANFILE are unaffected
MNTTRDB2 batch:
Given the DB2 transaction type table requires synchronization,
When COBTUPDT runs via MNTTRDB2 JCL,
Then DB2 table is updated and job completes with RC=0
