📋 Module Overview

Purpose: Backend batch jobs for transaction posting, interest calculation, statement generation, master file refresh, and data management. Batch processing is the backbone of end-of-day and end-of-month financial operations.

Business Context: Batch jobs run outside CICS hours (or with CICS files closed) to perform high-volume data processing. The nightly POSTTRAN and monthly INTCALC jobs are critical paths for financial accuracy.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Nightly posting: As a system operator, I want to run POSTTRAN nightly so account balances are updated with all processed transactions
  • Monthly interest: As a system, I want to run INTCALC monthly so interest is applied to all account balances before statements
  • Statement generation: As a system, I want to run CREASTMT monthly so all customers receive account statements
  • File refresh: As a system operator, I want to refresh master files so the test environment is reset to a known state
  • Batch window coordination: As a system operator, I want CLOSEFIL and OPENFIL to run reliably so CICS is not disrupted by batch operations
  • GDG versioning: As an auditor, I want report outputs stored in GDG so prior generations are available for comparison

⚡ Development Acceleration Factors

  • JCL templates: Existing JCL in app/jcl/ provides reusable DD statements and job control patterns
  • CLOSEFIL/OPENFIL pattern: Standard CICS file close/open already scripted — reuse for any new batch job touching CICS files
  • COBSWAIT utility: Timer control available for any batch sequence requiring delays
  • CSUTLDTC date utility: Date format conversion subroutine available — reuse in new batch programs needing date handling
  • GDG bases: DEFGDGB/DEFGDGD already define GDG bases — new reports can reference existing GDG structures

📊 Complexity Guidelines

Simple (1–2 pts) Change COBSWAIT delay parameter; modify JCL REGION/TIME limits; add new DD statement to existing job
Medium (3–5 pts) Add new step to existing JCL stream; modify POSTTRAN for new transaction category; add new batch report using existing TRANFILE data
Complex (5–8 pts) New batch program with VSAM RECLN change; parallel batch processing for performance; new GDG base definition and retirement policy; integrate new external data source into batch pipeline

🔧 Technical Foundation

Nightly Batch Sequence

1. COMBTRAN  — SORT combines transaction input files
2. CLOSEFIL  — Close CICS VSAM files (IEFBR14)
3. POSTTRAN  — CBTRN02C posts transactions, updates balances
4. TRANIDX   — IDCAMS rebuilds TRANFILE alternate index (if needed)
5. TRANREPT  — CBTRN03C generates daily report
6. OPENFIL   — Reopen CICS VSAM files (IEFBR14)

Monthly Batch Sequence

1. [Nightly sequence runs first]
2. CLOSEFIL  — Close CICS files
3. INTCALC   — CBACT04C applies interest to all accounts
4. CREASTMT  — CBSTM03A generates statements (CBSTM03B for detail)
5. TRANBKP   — IDCAMS backs up TRANFILE to GDG
6. TCATBALF  — IDCAMS resets category balances for new cycle
7. OPENFIL   — Reopen CICS files

Key Files and DD Names

ACCTFILE  → Account master VSAM KSDS
CARDFILE  → Card master VSAM KSDS
CUSTFILE  → Customer master VSAM KSDS
TRANFILE  → Transaction master VSAM KSDS
XREFFILE  → Cross-reference VSAM KSDS
TCATBALF  → Transaction category balance VSAM KSDS
DISCGRP   → Disclosure group (interest rates) VSAM KSDS
USRSEC    → User security VSAM KSDS

🔗 Dependencies

  • CLOSEFIL before update jobs: Any batch job that writes CICS-owned VSAM files must run after CLOSEFIL
  • OPENFIL after batch: CICS remains unavailable for VSAM-dependent transactions until OPENFIL completes
  • COMBTRAN before POSTTRAN: Transaction input must be sorted/combined before posting
  • INTCALC before CREASTMT: Interest must be applied before statements are generated
  • DEFGDGB before first report: GDG base definitions required before any GDG output jobs run

✅ Acceptance Criteria Examples

POSTTRAN success:
Given unposted transactions in TRANFILE,
When CBTRN02C completes with RC=0,
Then all transactions have TRAN-PROC-TS set; ACCT-CURR-BAL in ACCTFILE reflects all postings; TCATBALF updated with category totals
POSTTRAN — no transactions:
Given TRANFILE contains no unposted transactions,
When CBTRN02C runs,
Then it completes with RC=0 and "0 records processed" in job log
INTCALC balance update:
Given an account with category balance of $1,000 and interest rate of 1.5% in DISCGRP,
When CBACT04C runs,
Then ACCT-CURR-BAL is increased by $15.00 and INTCALC completes with RC=0
CLOSEFIL/OPENFIL coordination:
Given CICS is running with VSAM files open,
When CLOSEFIL step executes,
Then VSAM files are inaccessible to CICS transactions until OPENFIL completes; CICS programs receive appropriate error response code during this window
GDG versioning:
When CREASTMT writes statement output to a GDG dataset,
Then the new generation is accessible as (0); the previous generation is accessible as (-1); generation count does not exceed defined maximum
