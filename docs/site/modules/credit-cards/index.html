💳 credit-cards

CICS Online COBOL VSAM KSDS BMS 3270 Read / Write

The credit-cards module provides cardholders and administrators with the ability to list, view, and update credit card records stored in the CARDFILE VSAM dataset. It operates fully online via CICS 3270 terminal transactions and is accessed after successful account lookup in the accounts module.

1. Overview and Purpose

The credit-cards module manages the lifecycle of credit card records within the CardDemo application. It enables cardholders to browse all cards associated with their account, inspect full card details, and update mutable attributes such as the embossed name, active status, and expiration date. Admin users can additionally list cards across all accounts.

This module sits between the accounts module (which provides account context via COMMAREA) and the transactions module (which uses the card number as a cross-reference key for transaction lookups).

Module ID: credit-cards
Layer: Business Logic (CICS Online)
Primary VSAM File: CARDFILE (KSDS, RECLN=150)
Supporting VSAM File: XREFFILE (card-account-customer linkage)
CICS Transactions: CCLI, CCDL, CCUP

2. Architectural Context

CardDemo follows a classic mainframe OLTP architecture. Each CICS transaction consists of:

  • A BMS map — defines the 3270 terminal screen layout and field attributes
  • A COBOL program — contains all business logic, VSAM I/O, and navigation
  • A COMMAREA — the CARDDEMO-COMMAREA structure propagates session state (user ID, user type, current account/card) between transactions via CICS XCTL

The credit-cards module receives account context from the accounts module through the COMMAREA (CDEMO-ACCT-ID), uses that to filter CARDFILE reads, and passes the selected card number back into COMMAREA (CDEMO-CARD-NUM) for downstream transactions.

3. Key Components

COBOL Programs

COCRDLIC.cbl — Credit Card List (1,459 lines)

Transaction: CCLI  |  BMS Map: COCRDLI / CCRDLIAO

Displays up to 7 cards per page; supports PF7/PF8 pagination and single-key selection to view (S) or update (U) a card.
Admin behaviour: when CDEMO-USER-TYPE = 'A', no account filter is applied and cards from all accounts are listed. Regular users see only their account's cards.
Navigation targets: COCRDSLC (view), COCRDUPC (update), COMEN01C (main menu back).

COCRDSLC.cbl — Credit Card Detail / View (887 lines)

Transaction: CCDL  |  BMS Map: COCRDSL / CCRDSLAO

Reads and displays full card details for the account/card number combination received from COMMAREA. Read-only; does not modify CARDFILE.
Access path: reads by primary card number first; falls back to ACCT-PATH alternate index when only account ID is available.
Navigation targets: COCRDUPC (PF5 to update), COCRDLIC (PF3 back to list).

COCRDUPC.cbl — Credit Card Update (1,560 lines)

Transaction: CCUP  |  BMS Map: COCRDUP / CCRDUPAO

Full update of mutable card fields: embossed name, active status, and expiration date (month/year/day). Uses optimistic locking — detects concurrent changes between fetch and save.
Locking: EXEC CICS READ UPDATE acquires exclusive record lock; EXEC CICS REWRITE commits. Concurrent modification is detected by paragraph 9300-CHECK-CHANGE-IN-REC which compares the freshly locked DB values against the values displayed to the user at fetch time.
Navigation targets: COCRDSLC (PF3 cancel / post-save), COCRDLIC (back to list).

6. Business Rules and Validation Logic

6.1 Card Listing Rules (COCRDLIC)

  • Regular users (CDEMO-USER-TYPE = 'U'): list is filtered to cards belonging to the account in CDEMO-ACCT-ID.
  • Admin users (CDEMO-USER-TYPE = 'A'): may enter a free-form account number filter or browse all cards system-wide.
  • Maximum of 7 cards per page; PF7 scrolls backward, PF8 scrolls forward using VSAM BROWSE.
  • Selection column accepts only S (view) or U (update). Any other non-blank value is flagged as an error.
  • Only one card may be selected per submission. Multiple selections produce an error highlighting all affected rows.
  • When no cards exist for the account, a "no records found" message is shown and pagination keys are suppressed.

6.2 Card Detail Display Rules (COCRDSLC)

  • Displays data for the card identified by CDEMO-CARD-NUM in COMMAREA.
  • If card number is zero, falls back to the ACCT-PATH alternate index read using CDEMO-ACCT-ID.
  • Cross-reference validation against XREFFILE confirms the card-account linkage is intact.
  • Screen is read-only; no VSAM writes occur in this program.
  • PF5 transfers to COCRDUPC passing card context in COMMAREA.

6.3 Card Update Field Validation Rules (COCRDUPC)

  • Account Number: Must be numeric, non-zero, exactly 11 digits. Error: "Account number must be a non zero 11 digit number".
  • Card Number: If supplied, must be numeric and exactly 16 digits. Error: "Card number if supplied must be a 16 digit number".
  • Embossed Name: Must not be blank. Must contain only alphabetic characters and spaces. Error: "Card name can only contain alphabets and spaces".
  • Active Status: Must be Y or N. Error: "Card Active Status must be Y or N".
  • Expiry Month: Must be numeric in range 1–12. Error: "Card expiry month must be between 1 and 12".
  • Expiry Year: Must be numeric in range 1950–2099. Error: "Invalid card expiry year".
  • No change detected: If submitted values match current DB values, an informational message is shown and no write is performed. Message: "No change detected with respect to values fetched."
  • All validation errors are shown simultaneously on a single screen re-send — users see all errors at once.

6.4 Optimistic Concurrency Control

  • On PF5 (confirm save), the program issues EXEC CICS READ UPDATE to acquire an exclusive record lock on CARDFILE.
  • Paragraph 9300-CHECK-CHANGE-IN-REC compares the locked record fields against the values that were displayed to the user at initial fetch time (CCUP-OLD-* fields).
  • If any field differs (concurrent update by another session), the lock is released, the updated DB values are loaded into the screen, and the user is notified to review before saving again.
  • If lock acquisition itself fails (CICS RESP not NORMAL), COULD-NOT-LOCK-FOR-UPDATE error state is set and the operation is aborted without any write.
  • If the REWRITE fails after a successful lock, LOCKED-BUT-UPDATE-FAILED state is set and a failure message is displayed to the user.

6.5 Two-Phase Confirmation

  • Phase 1 (ENTER): Fields are validated. On all validations passing, screen message changes to "Changes validated. Press F5 to save". No write to CARDFILE occurs yet.
  • Phase 2 (PF5): Optimistic lock check runs; if no concurrent modification detected, EXEC CICS REWRITE commits the record. Success message: "Changes committed to database".

7. Processing Flows

7.1 Card List Flow (COCRDLIC)

  1. CICS invokes COCRDLIC; program checks COMMAREA for context and initial vs re-entry (CDEMO-PGM-CONTEXT).
  2. AID key is stored and validated. PF3 routes back to main menu via EXEC CICS XCTL PROGRAM(COMEN01C).
  3. For PF8 (next page) or initial entry: 9000-READ-FORWARD executes VSAM STARTBR on CARDFILE using CARD-ACCT-ID alternate index, then READNEXT up to 7 records.
  4. For PF7 (previous page): 9100-READ-BACKWARDS repositions the browse cursor and reads up to 7 records in reverse.
  5. Screen data array (7 rows) is populated and EXEC CICS SEND MAP renders the list.
  6. On re-entry, RECEIVE MAP captures user selection. Selection flags are validated (only 'S' or 'U', single selection only).
  7. Selected card number is stored in CDEMO-CARD-NUM in COMMAREA. Program XCTLs to COCRDSLC (S) or COCRDUPC (U).

7.2 Card Detail View Flow (COCRDSLC)

  1. Reads account ID and card number from COMMAREA.
  2. 9100-GETCARD-BYACCTCARD: attempts primary-key READ on CARDFILE; if card number is zero, falls back to AIX read using account ID path.
  3. Validates retrieved CARD-ACCT-ID matches COMMAREA account ID; sets error on mismatch.
  4. Formats CVV, card number, expiry date, and status into display-safe output fields.
  5. SEND MAP renders the read-only detail screen.
  6. PF5 XCTLs to COCRDUPC; PF3 returns to COCRDLIC.

7.3 Card Update Flow (COCRDUPC)

  1. Initial entry: reads card record (9000-READ-DATA) and populates screen with current values. Stores "old" field values in COMMAREA for later comparison.
  2. ENTER key: RECEIVE MAP captures edits. 1200-EDIT-MAP-INPUTS calls sub-paragraphs to validate account, card, name, status, month, and year. All errors surfaced simultaneously.
  3. On all validations passing, sets screen message to "Changes validated. Press F5 to save". Map re-sent — no write yet.
  4. PF5 key: 9200-WRITE-PROCESSING issues EXEC CICS READ FILE UPDATE to lock the record.
  5. 9300-CHECK-CHANGE-IN-REC compares locked record fields against the "old" values from step 1. If changed by another session, refreshes screen and notifies user.
  6. If no concurrent change: constructs CARD-UPDATE-RECORD merging new editable values with preserved CVV, then issues EXEC CICS REWRITE.
  7. On success, sets "Changes committed to database" and XCTLs back to COCRDSLC to show the refreshed card detail.
  8. PF3 cancel: XCTLs back to COCRDSLC without writing.

12. Code Examples

12.1 CARDFILE Browse with AIX (COCRDLIC — 9000-READ-FORWARD)

      * Reads up to 7 cards using CARD-ACCT-ID alternate index path
       9000-READ-FORWARD.
           EXEC CICS STARTBR
                FILE      (LIT-CARDFILENAME-ACCT-PATH)
                RIDFLD    (WS-CARD-RID-ACCT-ID)
                KEYLENGTH (LENGTH OF WS-CARD-RID-ACCT-ID)
                RESP      (WS-RESP-CD)
           END-EXEC

           MOVE 0 TO WS-ROWS-LOADED
           PERFORM UNTIL WS-ROWS-LOADED >= 7
                      OR READ-LOOP-EXIT
               EXEC CICS READNEXT
                    FILE      (LIT-CARDFILENAME-ACCT-PATH)
                    INTO      (CARD-RECORD)
                    RIDFLD    (WS-CARD-RID-ACCT-ID)
                    RESP      (WS-RESP-CD)
               END-EXEC
               IF WS-RESP-CD = DFHRESP(ENDFILE)
               OR WS-RESP-CD = DFHRESP(NOTFND)
                   SET READ-LOOP-EXIT TO TRUE
               ELSE
                   ADD 1 TO WS-ROWS-LOADED
                   * populate screen row WS-ROWS-LOADED from CARD-RECORD
               END-IF
           END-PERFORM
           EXEC CICS ENDBR FILE(LIT-CARDFILENAME-ACCT-PATH) END-EXEC.

12.2 Read Card by Primary Key or AIX (COCRDSLC — 9100-GETCARD-BYACCTCARD)

      * Attempts primary key read first; falls back to AIX when card num = 0
       9100-GETCARD-BYACCTCARD.
           IF CC-CARD-NUM-N NOT EQUAL ZEROS
               EXEC CICS READ
                    FILE      (LIT-CARDFILENAME)
                    INTO      (CARD-RECORD)
                    RIDFLD    (WS-CARD-RID-CARDNUM)
                    KEYLENGTH (LENGTH OF WS-CARD-RID-CARDNUM)
                    LENGTH    (LENGTH OF CARD-RECORD)
                    RESP      (WS-RESP-CD)
                    RESP2     (WS-REAS-CD)
               END-EXEC
           ELSE
               EXEC CICS READ
                    FILE      (LIT-CARDFILENAME-ACCT-PATH)
                    INTO      (CARD-RECORD)
                    RIDFLD    (WS-CARD-RID-ACCT-ID)
                    KEYLENGTH (LENGTH OF WS-CARD-RID-ACCT-ID)
                    LENGTH    (LENGTH OF CARD-RECORD)
                    RESP      (WS-RESP-CD)
                    RESP2     (WS-REAS-CD)
               END-EXEC
           END-IF
           IF WS-RESP-CD NOT EQUAL DFHRESP(NORMAL)
               MOVE 'READ'           TO ERROR-OPNAME
               MOVE LIT-CARDFILENAME TO ERROR-FILE
               MOVE WS-RESP-CD       TO ERROR-RESP
               MOVE WS-REAS-CD       TO ERROR-RESP2
               MOVE WS-FILE-ERROR-MESSAGE TO WS-RETURN-MSG
           END-IF.

12.3 Optimistic Lock and REWRITE (COCRDUPC — 9200-WRITE-PROCESSING)

      * Step 1: acquire exclusive lock
           EXEC CICS READ
                FILE      (LIT-CARDFILENAME)
                UPDATE
                RIDFLD    (WS-CARD-RID-CARDNUM)
                KEYLENGTH (LENGTH OF WS-CARD-RID-CARDNUM)
                INTO      (CARD-RECORD)
                LENGTH    (LENGTH OF CARD-RECORD)
                RESP      (WS-RESP-CD)
                RESP2     (WS-REAS-CD)
           END-EXEC
           IF WS-RESP-CD NOT EQUAL DFHRESP(NORMAL)
               SET INPUT-ERROR TO TRUE
               SET COULD-NOT-LOCK-FOR-UPDATE TO TRUE
               GO TO 9200-WRITE-PROCESSING-EXIT
           END-IF

      * Step 2: detect concurrent modification
           PERFORM 9300-CHECK-CHANGE-IN-REC THRU 9300-CHECK-CHANGE-IN-REC-EXIT
           IF DATA-WAS-CHANGED-BEFORE-UPDATE
               GO TO 9200-WRITE-PROCESSING-EXIT
           END-IF

      * Step 3: build update record
           STRING  CCUP-NEW-EXPYEAR '-' CCUP-NEW-EXPMON '-' CCUP-NEW-EXPDAY
                   DELIMITED BY SIZE INTO CARD-UPDATE-EXPIRAION-DATE
           MOVE CCUP-NEW-CRDNAME    TO CARD-UPDATE-EMBOSSED-NAME
           MOVE CCUP-NEW-CRDSTCD    TO CARD-UPDATE-ACTIVE-STATUS

      * Step 4: commit
           EXEC CICS
                REWRITE FILE(LIT-CARDFILENAME)
                        FROM(CARD-UPDATE-RECORD)
                        LENGTH(LENGTH OF CARD-UPDATE-RECORD)
                        RESP  (WS-RESP-CD)
                        RESP2 (WS-REAS-CD)
           END-EXEC
           IF WS-RESP-CD NOT EQUAL DFHRESP(NORMAL)
               SET LOCKED-BUT-UPDATE-FAILED TO TRUE
           END-IF.

12.4 Concurrent Modification Detection (COCRDUPC — 9300-CHECK-CHANGE-IN-REC)

      * Compares freshly-locked DB values against values shown to user at fetch time
       9300-CHECK-CHANGE-IN-REC.
           INSPECT CARD-EMBOSSED-NAME
              CONVERTING LIT-LOWER TO LIT-UPPER

           IF  CARD-CVV-CD              EQUAL TO CCUP-OLD-CVV-CD
           AND CARD-EMBOSSED-NAME       EQUAL TO CCUP-OLD-CRDNAME
           AND CARD-EXPIRAION-DATE(1:4) EQUAL TO CCUP-OLD-EXPYEAR
           AND CARD-EXPIRAION-DATE(6:2) EQUAL TO CCUP-OLD-EXPMON
           AND CARD-EXPIRAION-DATE(9:2) EQUAL TO CCUP-OLD-EXPDAY
           AND CARD-ACTIVE-STATUS       EQUAL TO CCUP-OLD-CRDSTCD
               CONTINUE                       * no concurrent change — safe to rewrite
           ELSE
               SET DATA-WAS-CHANGED-BEFORE-UPDATE TO TRUE
               MOVE CARD-CVV-CD              TO CCUP-OLD-CVV-CD
               MOVE CARD-EMBOSSED-NAME       TO CCUP-OLD-CRDNAME
               MOVE CARD-EXPIRAION-DATE(1:4) TO CCUP-OLD-EXPYEAR
               MOVE CARD-EXPIRAION-DATE(6:2) TO CCUP-OLD-EXPMON
               MOVE CARD-EXPIRAION-DATE(9:2) TO CCUP-OLD-EXPDAY
               MOVE CARD-ACTIVE-STATUS       TO CCUP-OLD-CRDSTCD
               GO TO 9200-WRITE-PROCESSING-EXIT
           END-IF.

12.5 Embossed Name Validation (COCRDUPC — 1230-EDIT-NAME)

      * Validates name is non-blank and contains only alpha characters and spaces
       1230-EDIT-NAME.
           MOVE CCUP-NEW-CRDNAME   TO CARD-NAME-CHECK
           INSPECT CARD-NAME-CHECK
              CONVERTING LOWER-CASE-CHARS TO UPPER-CASE-CHARS
           IF CARD-NAME-CHECK = LOW-VALUES OR SPACES
               SET INPUT-ERROR          TO TRUE
               SET WS-PROMPT-FOR-NAME   TO TRUE
           ELSE
               INSPECT CARD-NAME-CHECK
                  TALLYING WS-ALPHA-COUNT FOR ALL ALPHABETIC SPACES
               IF WS-ALPHA-COUNT = LENGTH OF CARD-NAME-CHECK
                   SET FLG-CARDNAME-ISVALID TO TRUE
               ELSE
                   SET INPUT-ERROR              TO TRUE
                   SET WS-NAME-MUST-BE-ALPHA    TO TRUE
               END-IF
           END-IF.

14. Development Guidelines

Adding a New Editable Field to COCRDUPC

  1. Add the field to the BMS map file COCRDUP.bms with an appropriate attribute byte (unprotected/alphabetic or numeric).
  2. Add corresponding CCUP-NEW-xxx and CCUP-OLD-xxx fields to the screen copybook COCRDUP.CPY.
  3. Add a flag variable (WS-EDIT-NEWFIELD-FLAG) with 88 FLG-NEWFIELD-ISVALID VALUE '1', FLG-NEWFIELD-NOT-OK VALUE '0', and FLG-NEWFIELD-BLANK VALUE ' '.
  4. Add an error message literal to the WS-RETURN-MSG 88-level values.
  5. Create a validation paragraph (1270-EDIT-NEWFIELD) following the existing pattern in COCRDUPC.
  6. Call the new paragraph from 1200-EDIT-MAP-INPUTS.
  7. Include the new field in 9300-CHECK-CHANGE-IN-REC comparison.
  8. Move the new field value into CARD-UPDATE-RECORD in 9200-WRITE-PROCESSING before the REWRITE.
  9. If the field requires a structural change to CVACT02Y.cpy, update CARDFILE RECLN in JCL, rebuild the VSAM cluster, reload data, and recompile all programs that COPY CVACT02Y.

Coding Conventions Observed in this Module

  • File names are stored as literals in WS-LITERAL-PARAMS: LIT-CARDFILENAME, LIT-CARDFILENAME-ACCT-PATH.
  • CICS RESP codes are checked after every EXEC CICS command without exception.
  • Input flags use the 88-level convention: FLG-xxx-ISVALID VALUE '1', FLG-xxx-NOT-OK VALUE '0', FLG-xxx-BLANK VALUE ' '.
  • Programs use GO TO COMMON-RETURN as the standard single exit point to prevent duplicate CICS command execution.
  • COMMAREA is declared as PIC X(2000) in LINKAGE SECTION; the first portion overlays CARDDEMO-COMMAREA and the remainder overlays program-local WS-THIS-PROGCOMMAREA.
  • Case-insensitive comparisons use INSPECT … CONVERTING LIT-LOWER TO LIT-UPPER before the comparison.

Known Limitations (Tech Debt)

  • CVV stored in plain text: CARD-CVV-CD is displayed and stored unencrypted. Intentional for the demo environment; must be masked or hashed before production use.
  • Expiry day not calendar-validated: Month (1–12) and year (1950–2099) are validated, but the day field is not checked against the given month/year combination.
  • Hardcoded program names: Navigation targets (COMEN01C, COCRDSLC, COCRDUPC) are compile-time constants; name changes require source edits and recompilation.
  • No delete operation: Card records cannot be deleted from CARDFILE through this module. Card deactivation (status='N') is the only retirement path available.
  • ACCT-PATH AIX dependency: The fallback read-by-account path depends on the CARD-ACCT-ID alternate index being defined and active on CARDFILE at all times.
