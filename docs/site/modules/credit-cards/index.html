💳 Credit Cards

User Story Development Guide — CardDemo Mainframe Application

CICS COBOL VSAM CARDFILE CCLI · CCDL · CCUP

📋 Module Overview

Purpose: List, view, and update credit card records linked to an account. Cards are the physical representation of account access; this module manages their details, status, and embossed information.

Business Context: Each account may have one or more credit cards. Cards are identified by a 16-digit PAN (Primary Account Number). The XREFFILE maintains the three-way link between card, account, and customer.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • List cards: As a cardholder, I want to see all cards on my account so I know which are active and which have expired
  • View card detail: As a cardholder, I want to view my card's expiration date and embossed name so I can verify my card information
  • Deactivate card: As a cardholder, I want to deactivate a lost or stolen card so unauthorized charges are prevented
  • Reactivate card: As a cardholder, I want to reactivate a previously deactivated card so I can resume using it
  • Update name: As a cardholder, I want to update the embossed name on my card so it reflects a legal name change

⚡ Development Acceleration Factors

  • CVACT02Y copybook: CARD-RECORD fully defined — field additions require only copybook and VSAM RECLN change
  • XREFFILE AIX: CARD-ACCT-ID alternate index already exists — account-to-card listing requires no new infrastructure
  • List/View/Update triad: Three programs follow the same VSAM read/write pattern — consistent to extend
  • PF key navigation: Standard CICS PF3/PF7/PF8 pattern shared with all other modules

📊 Complexity Guidelines

Simple (1–2 pts) Display additional card field read-only; change screen label text; add field to card detail view
Medium (3–5 pts) Add card status toggle with audit log; implement card expiry warning display; add card number masking (show last 4 only)
Complex (5–8 pts) Add new card field requiring VSAM RECLN change; implement card replacement flow (new card number, XREFFILE update); virtual card number generation

🔧 Technical Foundation

Card List Flow

Entry: ACCT-ID available in COMMAREA (set by Account module)
  → Read CARDFILE (AIX key = CARD-ACCT-ID) sequential
  → Collect cards matching current account
  → Display card list with pagination (PF7/PF8)
  → User selects card → passes CARD-NUM to COMMAREA
  → Navigate to Card Detail (CCDL)

Card Update Flow

Entry: CARD-NUM in COMMAREA from Card Detail screen
  → Read CARDFILE (primary key = CARD-NUM) for current values
  → Display on COCRDUP screen with editable fields
  → User modifies fields → ENTER
  → Validate inputs → REWRITE CARDFILE
  → Display success or error message

Data Model

CARD-RECORD (CARDFILE, RECLN=150):
  CARD-NUM              PIC X(16)  ← Primary key (16-digit PAN)
  CARD-ACCT-ID          PIC 9(11)  ← AIX key — links to ACCTFILE
  CARD-CVV-CD           PIC 9(03)
  CARD-EMBOSSED-NAME    PIC X(50)
  CARD-EXPIRAION-DATE   PIC X(10)  ← YYYY-MM-DD
  CARD-ACTIVE-STATUS    PIC X(01)  ← 'Y'=Active, 'N'=Inactive

CARD-XREF-RECORD (XREFFILE, RECLN=50):
  XREF-CARD-NUM         PIC X(16)  ← Primary key
  XREF-CUST-ID          PIC 9(09)  ← Customer ID
  XREF-ACCT-ID          PIC 9(11)  ← Account ID

🔗 Dependencies

  • accounts: ACCT-ID must be established in COMMAREA before card list can be filtered
  • CARDFILE VSAM: Primary card data store — CARD-ACCT-ID AIX must be defined for list function
  • XREFFILE VSAM: Three-way link table — must be consistent with CARDFILE and ACCTFILE
  • transactions: TRAN-CARD-NUM in TRANFILE references CARD-NUM — card deactivation does not auto-block existing transactions

✅ Acceptance Criteria Examples

Card list — shows all cards for account:
Given a signed-in cardholder with 2 cards on their account,
When navigating to Credit Card List (CCLI),
Then both card numbers (masked or full) with status and expiry are displayed
Card detail view:
Given a selected card number in COMMAREA,
When navigating to Card Detail (CCDL),
Then card number, embossed name, expiry date, CVV (or masked), and active status are displayed from CARDFILE
Deactivate card:
Given an active card (CARD-ACTIVE-STATUS = 'Y') on the update screen,
When changing status to 'N' and pressing ENTER,
Then CARDFILE is updated with CARD-ACTIVE-STATUS = 'N' and a confirmation message is shown
Update embossed name:
Given a valid new name (≤50 characters),
When submitting the update,
Then CARD-EMBOSSED-NAME in CARDFILE is updated and success message displayed
No cards on account:
Given an account with no associated cards in CARDFILE,
When navigating to card list,
Then "No cards found for this account" message is displayed
