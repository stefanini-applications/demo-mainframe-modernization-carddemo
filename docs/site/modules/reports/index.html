User Story Development Guide — CardDemo Mainframe Application

CICS Batch COBOL GDG Output Transaction: CR00

📋 Module Overview

Purpose: Generate and display transaction reports and account statements. Provides online report request (CORPT00C) and batch report generation (CBTRN03C for daily reports, CBSTM03A for statements).

Business Context: Reports provide financial visibility for cardholders, operators, and auditors. The daily transaction report (DALYREPT) is a key reconciliation tool. Account statements provide customers with billing summaries. Reports can be submitted from online (CICS) or run as standalone batch jobs.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Request report: As a cardholder, I want to request a transaction report for a date range so I can review my spending history
  • Daily reconciliation: As a system operator, I want to run the daily TRANREPT report so I can reconcile posted transaction totals
  • Monthly statement: As a system, I want to generate monthly statements for all accounts so customers receive billing summaries
  • Account summary: As a cardholder, I want to see page totals and account totals on the report so I can quickly find my total charges
  • Report history: As an auditor, I want access to prior report generations (via GDG) so I can compare month-over-month totals

⚡ Development Acceleration Factors

  • CVTRA07Y report structures: TRANSACTION-DETAIL-REPORT, page totals, account totals, grand total already defined
  • GDG output: Report versioning (multiple generations) already configured via DEFGDGB JCL — no new infrastructure for history
  • CICS-to-batch submission: CORPT00C pattern for submitting TRANREPT from online can be replicated for new report types
  • CBTRN03C reads TRANFILE: Same VSAM access pattern as transaction list — known and reliable

📊 Complexity Guidelines

Simple (1–2 pts) Add new column to existing report; change header text; modify sort order within existing CBTRN03C
Medium (3–5 pts) Add new report summary section (e.g. totals by category); add cardholder name to statement from CUSTFILE; implement date range filter for CREASTMT
Complex (5–8 pts) Entirely new report type (new COBOL program + JCL); PDF output via FTPJCL/TXT2PDF integration; online report view (display report data in CICS screen); multi-currency reporting

🔧 Technical Foundation

TRANREPT Flow (CBTRN03C)

Input: Start date, end date (from CORPT00C or JCL parameters)
  → Read TRANFILE sequentially
  → Filter by TRAN-ORIG-TS within date range
  → For each transaction:
      Look up type description (TRANTYPE file)
      Look up category description (TRANCATG file)
  → Group by Account ID
  → Write TRANSACTION-DETAIL-REPORT lines
  → Accumulate page totals, account totals, grand total
  → Write REPORT-PAGE-TOTALS, REPORT-ACCOUNT-TOTALS
  → Write REPORT-GRAND-TOTALS at end
  → Output to SYSOUT or GDG dataset

Report Output Structure

DALYREPT header line:
  Report Name (38) + Long Name (41) + Date Range header (12)
  + Start Date (10) + " to " + End Date (10)

Detail line (TRANSACTION-DETAIL-REPORT):
  Trans ID (16) | Account ID (11) | Type+Desc (17) |
  Cat+Desc (35) | Source (14) | Amount (signed, 16)

Totals lines:
  REPORT-PAGE-TOTALS    → "Page Total" + dots + signed amount
  REPORT-ACCOUNT-TOTALS → "Account Total" + dots + signed amount
  REPORT-GRAND-TOTALS   → "Grand Total" + dots + signed amount

🔗 Dependencies

  • transactions: TRANFILE is the primary data source for all reports
  • TRANTYPE VSAM: Transaction type descriptions used in report detail lines
  • TRANCATG VSAM: Transaction category descriptions used in report detail lines
  • CUSTFILE: Customer name used in statement generation (CREASTMT)
  • GDG setup: DEFGDGB JCL job must run before first report generation to define GDG bases

✅ Acceptance Criteria Examples

Report request from online:
Given a signed-in user on the Report screen (CR00),
When entering a valid start and end date and pressing ENTER,
Then TRANREPT job is submitted and a "Report submitted" message is displayed
Daily report content:
Given transactions exist within the specified date range,
When CBTRN03C completes with RC=0,
Then the output contains one detail line per transaction with correct type/category descriptions and amounts
Account totals accuracy:
Given multiple transactions for a single account,
When CBTRN03C produces account totals,
Then REPORT-ACCOUNT-TOTALS equals the algebraic sum of all TRAN-AMT values for that account
Grand total accuracy:
When CBTRN03C completes,
Then REPORT-GRAND-TOTALS equals the sum of all account totals in the report
No transactions in date range:
Given no transactions exist within the requested date range,
When CBTRN03C runs,
Then the report contains only headers and a grand total of $0.00 with a "No transactions found" indicator
