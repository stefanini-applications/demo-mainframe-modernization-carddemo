📊 Reports

User Story Development Guide — CardDemo Mainframe Application

CICS Batch COBOL VSAM TRANFILE GDG Output Transaction: CR00

📋 Module Overview

Purpose: Generate and display transaction reports and account statements. Provides an online report request screen (CORPT00C / transaction CR00) and two distinct batch report pipelines: the daily transaction detail report (CBTRN03C / TRANREPT job) and the full account statement generator (CBSTM03A + CBSTM03B / CREASTMT job).

Business Context: Reports provide financial visibility for cardholders, operators, and auditors. The daily transaction report (DALYREPT) is the primary reconciliation tool — it shows every posted transaction for a date range, grouped by account, with page, account, and grand totals. Account statements (CREASTMT) are the customer-facing billing documents: they include cardholder name, address, account balance, FICO score, and a transaction summary. Both plain-text and HTML-format statements are produced in a single batch run. Reports can be triggered from the CICS online screen (CR00) using an Extra Partition TDQ (JOBS), or submitted directly as standalone JCL jobs by operators.

🏗️ Architectural Context

The Reports module sits at the intersection of online CICS processing and backend batch execution. It demonstrates a key mainframe integration pattern: CICS-to-batch job submission via Extra Partition TDQ.

Online ↔ Batch Integration Pattern

CICS (CR00 / CORPT00C)
  │  User selects report type + date range
  │  CORPT00C builds inline JCL (hardcoded in WORKING STORAGE as JOB-DATA)
  │  Date parameters interpolated into JCL SYMNAMES and DATEPARM DD
  │  Each 80-byte JCL record written to Extra Partition TDQ "JOBS"
  ▼
z/OS Internal Reader (INTRDR) — reads from JOBS TDQ
  │  Receives JCL stream, submits as batch job
  ▼
TRANREPT Batch Job
  STEP05R: SORT TRANSACT VSAM → filter by date → TRANSACT.DALY GDG(+1)
  STEP10R: CBTRN03C reads sorted file → writes TRANREPT GDG(+1)

Module Dependency Diagram

reports
  ├── depends on: transactions (TRANFILE is primary data source)
  ├── depends on: accounts (ACCTFILE for statement balance/limit)
  ├── depends on: TRANTYPE VSAM (type descriptions in CBTRN03C)
  ├── depends on: TRANCATG VSAM (category descriptions in CBTRN03C)
  ├── depends on: CUSTFILE (cardholder name/address for statements)
  ├── depends on: XREFFILE (card→account→customer linkage for CREASTMT)
  └── consumed by: batch-processing (GDG reports, auditing)

GDG (Generation Data Groups)

TRANREPT GDG: AWS.M2.CARDDEMO.TRANREPT(+1) — each run creates a new generation, preserving all prior reports for audit.
TRANSACT.DALY GDG: AWS.M2.CARDDEMO.TRANSACT.DALY(+1) — daily extract of transactions sorted by card number, used as CBTRN03C input.
Statement output: AWS.M2.CARDDEMO.STATEMNT.PS (plain text) and AWS.M2.CARDDEMO.STATEMNT.HTML — recreated on each CREASTMT run.

CORPT00C Processing Logic

1
On first entry (CDEMO-PGM-REENTER = FALSE): display blank CORPT0A screen with cursor on MONTHLY field
2
User selects report type. For Monthly/Yearly: dates are auto-computed using FUNCTION CURRENT-DATE and date arithmetic. For Custom: user enters MM/DD/YYYY in start and end date fields.
3
On ENTER key: validate selected report type and date fields (all fields individually checked for spaces, numeric validity, range 1–12 months, 1–31 days). Date values validated via CALL 'CSUTLDTC' (date conversion utility).
4
If CONFIRM field is empty, prompt user with "Please confirm to print the [type] report..." message. User must enter Y or N.
5
On Y confirmation: PERFORM SUBMIT-JOB-TO-INTRDR. Iterates JOB-DATA array (up to 1000 × 80-byte JCL records), writing each to JOBS TDQ via EXEC CICS WRITEQ TD QUEUE('JOBS'). Stops at '/*EOF' sentinel.
6
On success: display green "[Monthly/Yearly/Custom] report submitted for printing..." message. Screen resets for next request.
7
PF3 → XCTL to COMEN01C (Main Menu). Any other key → invalid-key error message.

Inline JCL Construction

CORPT00C embeds the full TRANREPT JCL stream in JOB-DATA working storage. Date parameters are injected at runtime into two substitution points:

JOB-DATA working storage excerpt (CORPT00C lines 81-127):
  "//TRNRPT00 JOB 'TRAN REPORT',CLASS=A,MSGCLASS=0,"
  "// NOTIFY=&SYSUID"
  "//JOBLIB JCLLIB ORDER=('AWS.M2.CARDDEMO.PROC')"
  "//STEP10 EXEC PROC=TRANREPT"
  "//STEP05R.SYMNAMES DD *"
  "TRAN-CARD-NUM,263,16,ZD"
  "TRAN-PROC-DT,305,10,CH"
  PARM-START-DATE,C'[WS-START-DATE]'    ← date injected here
  PARM-END-DATE,C'[WS-END-DATE]'        ← date injected here
  "//STEP10R.DATEPARM DD *"
  [WS-START-DATE] [WS-END-DATE]         ← also injected here
  "/*EOF"
Note: The job name in the inline JCL is hardcoded as TRNRPT00, not the same as the standalone TRANREPT job. Both invoke the same TRANREPT procedure from AWS.M2.CARDDEMO.PROC.

Processing Flow

PROCEDURE DIVISION:
  0000-TRANFILE-OPEN      → open TRANSACT.DALY sequential input
  0100-REPTFILE-OPEN      → open TRANREPT GDG(+1) output
  0200-CARDXREF-OPEN      → open CARDXREF VSAM
  0300-TRANTYPE-OPEN      → open TRANTYPE VSAM
  0400-TRANCATG-OPEN      → open TRANCATG VSAM
  0500-DATEPARM-OPEN      → open DATEPARM
  0550-DATEPARM-READ      → read start/end dates into WS-START-DATE, WS-END-DATE

  MAIN LOOP (PERFORM UNTIL END-OF-FILE = 'Y'):
    1000-TRANFILE-GET-NEXT
    Filter: TRAN-PROC-TS(1:10) must be >= WS-START-DATE AND <= WS-END-DATE
    On new card number:
      If not first time: 1120-WRITE-ACCOUNT-TOTALS
      1500-A-LOOKUP-XREF     → get Account ID from XREFFILE
    1500-B-LOOKUP-TRANTYPE   → get type description
    1500-C-LOOKUP-TRANCATG   → get category description (key: type-cd + cat-cd)
    1100-WRITE-TRANSACTION-REPORT
      Move fields to TRANSACTION-DETAIL-REPORT from CVTRA07Y
      Write 133-char line to TRANREPT
      If WS-LINE-COUNTER >= WS-PAGE-SIZE (20 lines/page):
        1110-WRITE-PAGE-TOTALS

  End of file:
    1120-WRITE-ACCOUNT-TOTALS (last account)
    1110-WRITE-GRAND-TOTALS

  9000–9500: Close all files

Date Filtering Detail

CBTRN03C filters on TRAN-PROC-TS(1:10) (processing timestamp, first 10 characters = YYYY-MM-DD). The TRANREPT JCL SORT step pre-filters using INCLUDE COND on TRAN-PROC-DT (field at offset 305, length 10), so CBTRN03C operates on a pre-reduced dataset. Both filter conditions use the same date range from DATEPARM.

📄 Batch Component: CBSTM03A / CBSTM03B (CREASTMT Job)

CBSTM03A — Statement Orchestrator

CBSTM03A generates full account statements by iterating over all card+transaction combinations in the TRNXFILE VSAM (which CREASTMT.JCL pre-builds from TRANSACT by re-keying on card-number + tran-id). For each unique card, it retrieves customer and account data, formats statement lines, and writes both plain-text and HTML outputs.

📏 Business Rules

  • Date range is mandatory for TRANREPT: All three report types (Monthly, Yearly, Custom) require a valid start and end date. Monthly/Yearly dates are auto-computed; Custom requires explicit MM/DD/YYYY entry.
  • Custom date validation: Each date component (month, day, year) is validated individually — must be numeric, month 01–12, day 01–31, year 4 digits. Additionally, the date is validated as a calendar-valid date via CALL 'CSUTLDTC'.
  • Confirmation required: The report is not submitted until the user explicitly enters 'Y' in the CONFIRM field. 'N' cancels and resets the screen. Any other value produces an error.
  • TDQ write failure stops submission: If EXEC CICS WRITEQ TD to JOBS TDQ returns non-NORMAL response, the submission is aborted and an error message is displayed. Partial JCL already written to TDQ is not retracted.
  • TRANREPT filters on processing timestamp: CBTRN03C uses TRAN-PROC-TS(1:10) for date range filtering — the processing date, not the original transaction date. Ensure the date range covers the posting/processing run date, not the purchase date.
  • Page size fixed at 20 lines: CBTRN03C writes page totals after every 20 transaction detail lines (WS-PAGE-SIZE = 20).
  • Account grouping depends on sort order: CBTRN03C groups transactions by card number (detects card number changes). Transactions must be pre-sorted by card number via the TRANREPT JCL SORT step before CBTRN03C runs.
  • CREASTMT processes all accounts: No date range filter — all transactions in TRNXFILE are included in the statement. CREASTMT.JCL does not parameterize dates; every run produces statements for all card/transaction data in the VSAM.
  • Statement outputs are overwritten each run: STEP030 of CREASTMT.JCL deletes prior STATEMNT.HTML and STATEMNT.PS before generating new ones. No GDG versioning for statement files.
  • GDG base must exist: TRANREPT GDG base (AWS.M2.CARDDEMO.TRANREPT) must be defined via DEFGDGB JCL before the first TRANREPT run. Failure to define the base causes the job to abend on GDG allocation.
  • CORPT00C requires COMMAREA: If EIBCALEN = 0 (no COMMAREA), CORPT00C transfers control back to COSGN00C (signon). Users must be authenticated before accessing reports.

🎯 User Story Development Guide

📝 Module-Specific Story Templates

  • Request monthly report: As a cardholder, I want to request a monthly transaction report from the CICS screen (CR00) so I can review my spending for the current month
  • Request custom date range: As a cardholder, I want to enter a custom date range (MM/DD/YYYY) and receive a filtered transaction report so I can investigate charges in a specific period
  • Operator daily reconciliation: As a system operator, I want to run the TRANREPT batch job (CBTRN03C) for yesterday's date range so I can reconcile posted transaction totals
  • Monthly statement generation: As a system administrator, I want to run the CREASTMT job so all customers receive billing summaries in both plain-text and HTML formats
  • Report totals verification: As an auditor, I want page totals, account totals, and a grand total on the TRANREPT report so I can quickly verify the sum of all charges
  • Historical report access: As an auditor, I want to access prior report generations via GDG so I can compare month-over-month transaction volumes
  • Statement content review: As a cardholder, I want my statement to show my current balance, FICO score, and all transactions so I have a complete billing summary

⚡ Development Acceleration Factors

  • CVTRA07Y report structures: TRANSACTION-DETAIL-REPORT, REPORT-PAGE-TOTALS, REPORT-ACCOUNT-TOTALS, and REPORT-GRAND-TOTALS are all pre-defined — report field layout changes require only copybook modification
  • GDG output already configured: Report versioning (multiple generations) is established via DEFGDGB JCL — no new infrastructure needed for historical report retention
  • CICS-to-batch TDQ pattern: The CORPT00C inline JCL submission pattern can be replicated for any new batch report type without changing the CICS infrastructure
  • Date arithmetic helpers: CORPT00C already demonstrates FUNCTION CURRENT-DATE, FUNCTION DATE-OF-INTEGER, and FUNCTION INTEGER-OF-DATE for computing month/year start and end dates
  • Dual output already built: CBSTM03A produces both text and HTML in a single pass — extending to new output formats (CSV, JSON) requires only new HTML-LINES-style data structures
  • CBSTM03B abstraction: The I/O subroutine pattern isolates file handling — adding a new file to statements requires only a new WHEN branch in CBSTM03B and corresponding records in CBSTM03A

Simple (1–2 pts)
  • Add/change column in TRANSACTION-DETAIL-REPORT layout
  • Modify report header text or page size (WS-PAGE-SIZE)
  • Change sort order within existing CBTRN03C
  • Add bank name/address to statement header
Medium (3–5 pts)
  • Add new totals section (e.g. totals by transaction category)
  • Add date range parameter to CREASTMT
  • Merchant name/city on TRANREPT detail line
  • New report type on CORPT00C screen (additional radio option)

Complex (5–8 pts)
  • Entirely new batch report program + JCL + GDG definition
  • Online report viewer (display TRANREPT output in CICS)
  • CSV/JSON export output format
  • Multi-currency reporting with exchange rates
  • Report delivery via MQ or email gateway

✅ Acceptance Criteria Examples

Online report request — Monthly:
Given an authenticated user on CR00 screen,
When selecting "Monthly" and pressing ENTER without entering a confirmation,
Then the screen prompts "Please confirm to print the Monthly report..." with a Y/N field.
When entering 'Y' and pressing ENTER,
Then JCL is written to JOBS TDQ and the screen shows "Monthly report submitted for printing..." in green.
Custom date validation:
Given the user selects "Custom" date range,
When entering month 13 in SDTMM,
Then an error "Start Date - Not a valid Month..." is displayed in red on line 23 and the cursor returns to SDTMM.
When entering an invalid calendar date (e.g. 02/30/2022),
Then CSUTLDTC returns a non-zero severity and "Start Date - Not a valid date..." is displayed.
TRANREPT batch content:
Given transactions exist with TRAN-PROC-TS within the specified date range,
When CBTRN03C completes with RC=0,
Then the report GDG contains one detail line per in-scope transaction,
And each line includes the correct type description (from TRANTYPE) and category description (from TRANCATG),
And amounts are formatted as -ZZZ,ZZZ,ZZZ.ZZ with correct sign.
Account total accuracy:
Given N transactions for card number C within the date range,
When CBTRN03C processes the last transaction for C and switches to the next card,
Then REPORT-ACCOUNT-TOTALS = algebraic sum of all TRAN-AMT values for C,
And WS-ACCOUNT-TOTAL is reset to 0 for the next account group.
Grand total integrity:
When CBTRN03C completes,
Then REPORT-GRAND-TOTALS equals the sum of all REPORT-ACCOUNT-TOTALS in the report,
And the GDG allocation is successful (new generation created, RC=0).
Statement dual output:
When CREASTMT job completes with RC=0,
Then AWS.M2.CARDDEMO.STATEMNT.PS exists and contains plain-text statements for all accounts in TRNXFILE,
And AWS.M2.CARDDEMO.STATEMNT.HTML exists and contains valid HTML with <!DOCTYPE html> and table layout for all accounts,
And each statement includes account ID, current balance, FICO score, and a transaction summary section.
No authentication — redirect to signon:
Given a user accesses CR00 without a valid COMMAREA (EIBCALEN = 0),
When CORPT00C executes,
Then the program performs XCTL to COSGN00C without displaying the report screen.

⚠️ Known Issues and Tech Debt

  • Hardcoded job name: The CICS-submitted job is named TRNRPT00 (embedded in CORPT00C JOB-DATA working storage). Changing the job name requires a recompile of CORPT00C.
  • No online report viewer: TRANREPT output is batch-only; there is no CICS screen to display the report. A modernization effort would add an online view transaction that reads the GDG output and paginates it on 3270.
  • Statement GDG not used: Unlike TRANREPT, statement output (STATEMNT.PS and STATEMNT.HTML) overwrites the previous run — no GDG versioning for customer statements. History is lost on each CREASTMT run.
  • No statement date range: CREASTMT processes all transactions regardless of date. There is no parameter mechanism to restrict statements to a billing period in the current implementation.
  • Partial TDQ writes on error: If CORPT00C encounters a TDQ write error mid-stream, partial JCL lines may already be in the JOBS queue. The internal reader may attempt to submit a malformed job. Error recovery is limited to the displayed error message.
  • CBSTM03A uses ALTER/GO TO: The program intentionally demonstrates legacy COBOL patterns (ALTER and GO TO statements) as a modernization exercise target. These constructs complicate static analysis and refactoring.
  • HTML bank name hardcoded: CBSTM03A HTML output contains hardcoded "Bank of XYZ" and "410 Terry Ave N, Seattle WA 99999" — not driven by any configurable parameter.
