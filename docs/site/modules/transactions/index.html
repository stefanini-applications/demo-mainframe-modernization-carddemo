CICS Online Batch JCL VSAM TRANFILE VSAM TCATBALF CT00 · CT01 · CT02 POSTTRAN · TRANREPT

Overview

The transactions module provides full lifecycle management of credit card transactions within the CardDemo mainframe application. It encompasses three CICS online screens for listing, viewing, and adding transactions; three batch programs for validation, posting, and reporting; and a suite of JCL jobs that orchestrate batch processing. All transaction data is persisted in the VSAM KSDS file TRANFILE with a 16-character transaction ID as the primary key and an Alternate Index (AIX) on card number for efficient per-card lookups.

Module ID: transactions

Processing Flows

CT00 — Transaction List (COTRN00C)

1. On first entry: read COMMAREA to get CDEMO-CARD-NUM (current card context)
2. Execute EXEC CICS STARTBR DATASET('TRANSACT') RIDFLD(TRAN-CARD-NUM) GTEQ on the AIX
3. Loop READNEXT up to 10 records for the current page; track first/last TRAN-ID in COMMAREA (CDEMO-CT00-TRNID-FIRST, CDEMO-CT00-TRNID-LAST)
4. Peek at the (N+1)th record to set CDEMO-CT00-NEXT-PAGE-FLG; execute ENDBR
5. PF8 (forward): restart STARTBR at TRNID-LAST, skip one, read next 10
6. PF7 (backward): STARTBR at TRNID-FIRST, loop READPREV 10 records
7. 'S' selection: XCTL to COTRN01C with selected TRAN-ID in CDEMO-CT00-TRN-SELECTED

CT01 — Transaction View (COTRN01C)

1. On entry from CT00: CDEMO-CT01-TRN-SELECTED contains the TRAN-ID to display
2. Execute EXEC CICS READ DATASET('TRANSACT') INTO(TRAN-RECORD) RIDFLD(TRAN-ID) UPDATE
3. Format TRAN-AMT as +99999999.99 and populate all BMS map output fields
4. ENTER key: re-execute READ for the TRAN-ID entered in the input field
5. PF5: XCTL back to COTRN00C (transaction list)
6. PF3: XCTL to COMEN01C or the previous program from COMMAREA

CT02 — Add Transaction (COTRN02C)

1. User enters Account ID or Card Number (one field populates the other via XREFFILE lookup)
2. Validate: type code, category code, amount, merchant fields are all non-blank/valid
3. MOVE HIGH-VALUES TO TRAN-ID → STARTBR → READPREV → ENDBR to get the last TRAN-ID
move back to TRAN-ID (character)
5. Build TRAN-RECORD from screen input; set TRAN-ORIG-TS and TRAN-PROC-TS from current timestamp
6. EXEC CICS WRITE DATASET('TRANSACT') FROM(TRAN-RECORD) RIDFLD(TRAN-ID)
7. On success: display confirmation; on DUPREC: regenerate ID and retry

POSTTRAN Batch (CBTRN02C)

1. Open all files: DALYTRAN (input), TRANFILE (VSAM), XREFFILE, DALYREJS (reject output), ACCTFILE, TCATBALF
2. Loop: READ DALYTRAN-FILE INTO DALYTRAN-RECORD until end-of-file
3. 1500-VALIDATE-TRAN: READ XREFFILE by DALYTRAN-CARD-NUM → get XREF-ACCT-ID; READ ACCTFILE by ACCT-ID → check credit limit (ACCT-CURR-CYC-CREDIT)
4. 2000-POST-TRANSACTION: generate DB2 timestamp; call 2700-UPDATE-TCATBAL; call 2800-UPDATE-ACCOUNT-REC; call 2900-WRITE-TRANSACTION-FILE
5. 2700-UPDATE-TCATBAL: READ-UPDATE TCATBALF; ADD DALYTRAN-AMT TO TRAN-CAT-BAL; REWRITE
6. 2800-UPDATE-ACCOUNT-REC: ADD DALYTRAN-AMT TO ACCT-CURR-BAL; REWRITE ACCTFILE record
7. 2900-WRITE-TRANSACTION-FILE: WRITE TRANFILE record; on duplicate: increment ID and retry
8. On validation failure: WRITE reject record to DALYREJS; increment WS-REJECT-COUNT
9. DISPLAY reject count; if > 0 set RC=8

Business Rules

Transaction ID Generation: COTRN02C generates the next TRAN-ID by reading the last record in TRANFILE (STARTBR HIGH-VALUES + READPREV), converting to numeric, adding 1, and converting back. This is a sequential counter in VSAM key order.
Amount Sign Convention: Negative amounts (TRAN-AMT < 0) represent charges/debits (reduce account balance). Positive amounts represent credits/payments (increase account balance). The POSTTRAN job adds DALYTRAN-AMT directly to ACCT-CURR-BAL.
Card-Account Linkage: COTRN02C accepts either an Account ID or Card Number. If Account ID is entered, it reads CCXREF using the AIX (XREF-ACCT-ID) to find the card number. If Card Number is entered, it reads CCXREF by primary key to find the Account ID. Both lookups use EXEC CICS READ DATASET('CCXREF').
Transaction List Scope: CT00 (COTRN00C) lists transactions for the card number stored in CDEMO-CARD-NUM in the COMMAREA. The TRANFILE AIX on TRAN-CARD-NUM is used for the browse (STARTBR). Transactions for other cards are not visible.
Category Balance Update: POSTTRAN (CBTRN02C) maintains the TCATBALF file, which stores running balances per Account+TypeCode+CategoryCode. These are read by the INTCALC batch job (CBACT04C) together with DISCGRP interest rates to compute monthly interest per group.
Reject Handling: Transactions that fail POSTTRAN validation (invalid card in XREFFILE, or account not found) are written to the DALYREJS GDG dataset (LRECL=430, includes reject reason trailer). The batch job returns RC=8 when any rejects exist.
Date Range Filtering: The TRANREPT job uses a SORT INCLUDE on TRAN-PROC-TS(305,10) against date parameters from the DATEPARM file. The date format is YYYY-MM-DD (character comparison). The CORPT00C online screen allows the user to provide the date range which is written to the TDQ.
CICS File Close Requirement: POSTTRAN (CBTRN02C) directly updates TRANFILE, ACCTFILE, and TCATBALF. These VSAM files must be closed in CICS via the CLOSEFIL job before POSTTRAN runs, and reopened via OPENFIL after completion. Running POSTTRAN while CICS has the files open risks data corruption and VSAM file status errors.
No Credit-Limit Enforcement at Add-Time: COTRN02C does not check the credit limit before writing a transaction. Credit limit validation occurs in CBTRN02C (POSTTRAN batch) during the 1500-VALIDATE-TRAN paragraph. Online transaction adds can therefore temporarily exceed the credit limit until batch posting runs.

Usage Examples

Example 1: CICS Browse (COTRN00C excerpt)

       STARTBR-TRANSACT-FILE.
           EXEC CICS STARTBR
                DATASET   (WS-TRANSACT-FILE)      *> 'TRANSACT'
                RIDFLD    (TRAN-CARD-NUM)          *> Start at card number (AIX)
                GTEQ
                RESP      (WS-RESP-CD)
                RESP2     (WS-REAS-CD)
           END-EXEC.

       READNEXT-TRANSACT-FILE.
           EXEC CICS READNEXT
                DATASET   (WS-TRANSACT-FILE)
                INTO      (TRAN-RECORD)
                LENGTH    (LENGTH OF TRAN-RECORD)
                RIDFLD    (TRAN-ID)               *> Updated by CICS on each read
                RESP      (WS-RESP-CD)
                RESP2     (WS-REAS-CD)
           END-EXEC.
           EVALUATE WS-RESP-CD
               WHEN DFHRESP(NORMAL)   CONTINUE
               WHEN DFHRESP(ENDFILE)  SET TRANSACT-EOF TO TRUE
               WHEN OTHER             SET ERR-FLG-ON TO TRUE
           END-EVALUATE.

Example 2: Write New Transaction (COTRN02C excerpt)

       ADD-TRANSACTION.
           *> Step 1: Determine next TRAN-ID
           MOVE HIGH-VALUES TO TRAN-ID
           PERFORM STARTBR-TRANSACT-FILE
           PERFORM READPREV-TRANSACT-FILE      *> Gets the last record
           PERFORM ENDBR-TRANSACT-FILE
           MOVE TRAN-ID     TO WS-TRAN-ID-N   *> Convert X(16) to 9(16)
           ADD 1 TO WS-TRAN-ID-N
           MOVE WS-TRAN-ID-N TO TRAN-ID       *> New sequential key

           *> Step 2: Build record from screen input
           MOVE CARDNINI  OF COTRN2AI TO TRAN-CARD-NUM
           MOVE TTYPCDI   OF COTRN2AI TO TRAN-TYPE-CD
           MOVE TCATCDI   OF COTRN2AI TO TRAN-CAT-CD
           MOVE TRNAMTI   OF COTRN2AI TO TRAN-AMT
           MOVE FUNCTION CURRENT-DATE  TO WS-TIMESTAMP
           MOVE WS-TIMESTAMP TO TRAN-ORIG-TS
                               TRAN-PROC-TS

           *> Step 3: Write to VSAM
           EXEC CICS WRITE
                DATASET   (WS-TRANSACT-FILE)
                FROM      (TRAN-RECORD)
                LENGTH    (LENGTH OF TRAN-RECORD)
                RIDFLD    (TRAN-ID)
                RESP      (WS-RESP-CD)
           END-EXEC.

Example 3: POSTTRAN Batch Balance Update (CBTRN02C excerpt)

       2800-UPDATE-ACCOUNT-REC.
           *> Read account with update lock
           READ ACCOUNT-FILE INTO ACCOUNT-RECORD
               KEY IS WS-ACCT-ID
               INVALID KEY
                   DISPLAY 'ACCOUNT NOT FOUND: ' WS-ACCT-ID
                   PERFORM 9999-ABEND-PROGRAM
           END-READ.

           *> Apply transaction amount to balance
           ADD DALYTRAN-AMT TO ACCT-CURR-BAL      *> Negative for charges, positive for credits

           *> Update debit/credit cycle accumulators
           IF DALYTRAN-AMT < ZEROS
               ADD DALYTRAN-AMT TO ACCT-CURR-CYC-DEBIT
           ELSE
               ADD DALYTRAN-AMT TO ACCT-CURR-CYC-CREDIT
           END-IF.

           REWRITE FD-ACCTFILE-REC FROM ACCOUNT-RECORD
               INVALID KEY
                   DISPLAY 'ERROR REWRITING ACCOUNT: ' WS-ACCT-ID
                   PERFORM 9999-ABEND-PROGRAM
           END-REWRITE.

Development Guidelines

Patterns Used in This Module

  • Pseudo-conversational CICS: Each screen interaction ends with EXEC CICS RETURN TRANSID(WS-TRANID) COMMAREA(CARDDEMO-COMMAREA) — state is carried in COMMAREA, not in working storage across interactions.
  • COMMAREA Extension Fields: CT00 and CT01 each add a private navigation block to COMMAREA (CDEMO-CT00-INFO, CDEMO-CT01-INFO) to track pagination state and selected transaction IDs.
  • GTEQ Browse: AIX browse starts at or after the card number using GTEQ. Programs loop READNEXT until the TRAN-CARD-NUM in the returned record no longer matches the target card number.
  • EIBCALEN Guard: All CICS programs check IF EIBCALEN = 0 at entry and redirect to COSGN00C if no COMMAREA is present — preventing direct transaction invocation without a session.
  • Error Flag Pattern: WS-ERR-FLG (88-level ERR-FLG-ON) is set on any error; downstream paragraphs check this flag before proceeding.
  • High-Values ID Generation: CT02 uses MOVE HIGH-VALUES to TRAN-ID before STARTBR to position at the end of the KSDS, then READPREV to get the last record.

Tech Debt & Modernisation Notes

  • Sequential ID generation: Using STARTBR/READPREV to generate the next transaction ID is not concurrency-safe under high CICS transaction volumes. A modernised design should use a DB2 sequence or UUID.
  • No online credit-limit check: CT02 does not enforce credit limits; validation only occurs during POSTTRAN batch. Consider adding an online check against ACCTFILE during CT02 validation.
  • CICS file close dependency: Batch jobs updating TRANFILE require CLOSEFIL/OPENFIL coordination. A modernised design would use VSAM RLS (Record Level Sharing) to allow concurrent CICS and batch access.
  • Plain-text timestamps: TRAN-ORIG-TS / TRAN-PROC-TS are PIC X(26) character strings. A modernised DB would use native TIMESTAMP types.
