Overview

The Authorization module is an optional extension to CardDemo that implements end-to-end credit card authorization processing. It integrates three enterprise technologies: IMS hierarchical database, DB2 relational database, and IBM MQ message queuing, alongside the base VSAM files used by the core application.

Authorization requests originate from external POS systems (or any MQ-compatible client) and arrive asynchronously via an MQ input queue. A CICS-triggered program validates the request against account and card data, applies business rules (credit availability, card status, fraud flags), and returns an approve/decline response via an MQ reply queue. Approved and declined records are persisted in IMS DB for operational review, and suspected fraud cases are escalated to a DB2 table for analytics.

Location: app/app-authorization-ims-db2-mq/
Prerequisites: Base CardDemo application must be fully deployed before installing this module. IMS DB, DB2, and IBM MQ must be available and configured in the target CICS region.

Key Capabilities

  • Real-time credit card authorization processing via MQ message trigger
  • Authorization request validation with multi-condition business rule evaluation
  • Hierarchical storage of authorization records in IMS DB (summary + detail segments)
  • Fraud detection and reporting with DB2 integration for analytics
  • Two-phase operation across IMS DB and DB2 within a CICS unit of work
  • Interactive authorization history viewing (summary and detail 3270 screens)
  • Batch purging of expired pending authorizations with credit balance adjustment

Architecture

The authorization module follows an event-driven architecture layered on top of the CardDemo CICS transaction processing platform. The core flow is MQ-triggered, while administrative screens use conventional 3270 terminal interactions.

Key Components

CICS COPAUA0C.cbl

Transaction: CP00

Authorization request processor. Triggered by MQ message arrival. Reads up to 500 messages per invocation from the request queue, validates each request against VSAM account/card data, applies business rules, inserts/updates IMS records, and sends approve/decline responses to the MQ reply queue.

CICS COPAUS0C.cbl

Transaction: CPVS   Screen: COPAU00

Pending Authorization Summary display. Reads the IMS summary segment (PAUTSUM0) for the current account and shows aggregate statistics: approved/declined counts and amounts, credit limit, and cash limit.

CICS COPAUS1C.cbl

Transaction: CPVD   Screen: COPAU01

Pending Authorization Details display. Reads the IMS detail segments (PAUTDTL1) for a specific authorization. Allows operators to mark transactions as fraudulent. Calls COPAUS2C to insert fraud records into DB2 and update the IMS segment.

CICS COPAUS2C.cbl

Called by: COPAUS1C (EXEC CICS LINK)

Fraud marking and DB2 update program. Inserts a fraud record into the DB2 AUTHFRDS table and updates the corresponding IMS PAUTDTL1 segment with fraud flag and report date. DB2 schema name must be configured for the target environment.

Batch CBPAUP0C.cbl

Job: CBPAUP0J

Expired authorization purge batch program. Uses IMS BMP interface (PSBPAUTB) to scan all PAUTSUM0 root segments and delete PAUTDTL1 detail segments older than a configurable expiry threshold (P-EXPIRY-DAYS PARM). Adjusts credit balance and issues IMS checkpoints for restart capability.

DBUNLDGS.CBL / PAUDBLOD.CBL / PAUDBUNL.CBL

Utility programs for IMS database unload and reload operations. Used for initial data loading and database maintenance via JCL jobs UNLDGSAM, DBPAUTP0, UNLDPADB, and LOADPADB.

Business Rules

Authorization Decision Logic (COPAUA0C)

Rules are evaluated in order. The first failing condition causes a decline:

#RuleConditionDecline Flag
1Card Fraud CheckIMS summary fraud flag set for cardCARD-FRAUD
2Merchant Fraud CheckMerchant is fraud-flagged in IMSMERCHANT-FRAUD
3Account Active CheckACCT-ACTIVE-STATUS ≠ 'Y' in ACCTFILEACCOUNT-CLOSED
4Card Active CheckCARD-ACTIVE-STATUS ≠ 'Y' in CARDFILECARD-NOT-ACTIVE
5IMS Credit CheckTransaction amount > PA-CREDIT-LIMIT (IMS)INSUFFICIENT-FUND
6VSAM Credit CheckTransaction amount > ACCT-CREDIT-LIMIT (ACCTFILE fallback)INSUFFICIENT-FUND

Processing Limits and Thresholds

  • Maximum MQ messages per CP00 invocation: 500 (WS-REQSTS-PROCESS-LIMIT)
  • MQ reply message expiry: 50 MQ time units (MQMD-EXPIRY)
  • IMS checkpoint frequency: Controlled by WK-CHKPT-ID-CTR in batch
  • Batch expiry threshold: P-EXPIRY-DAYS (2-digit PARM parameter, e.g., 30 days)

Fraud Rules

  • Fraud is marked by an operator via the CPVD screen (COPAUS1C)
  • COPAUS2C inserts a record into DB2 AUTHFRDS and sets PA-AUTH-FRAUD = 'F' in IMS
  • Fraud report date stored in PA-FRAUD-RPT-DATE (YYYYMMDD)
  • Fraud can be removed by setting PA-AUTH-FRAUD = 'R'
  • Future authorization requests from a fraud-flagged card are declined with reason CARD-FRAUD

IMS Update on Authorization

  • On approval: increment PA-APPROVED-AUTH-CNT, add to PA-APPROVED-AUTH-AMT, reduce PA-CREDIT-BALANCE by approved amount
  • On decline: increment PA-DECLINED-AUTH-CNT, add to PA-DECLINED-AUTH-AMT
  • PAUTSUM0 root updated via IMS REPL; new PAUTDTL1 child inserted via IMS ISRT
  • If no PAUTSUM0 root exists for account, a new root segment is created before detail insert

Processing Flow

Authorization Request Flow (CP00 / COPAUA0C)

1 MQ message arrives on AWS.M2.CARDDEMO.PAUTH.REQUEST; CICS trigger fires CP00 transaction
2 COPAUA0C opens MQ request queue (MQOPEN) and reply queue; saves correlation ID
3 Schedule IMS PSB PSBPAUTB via EXEC CICS DLI SCHD PSB; set PCB offset to +1
4 Loop — MQGET next message from request queue (up to 500 messages):
   a) Parse CSV fields into CCPAURQY request structure
   b) Look up card in VSAM XREFFILE (CCXREF AIX by card number) to get account ID and customer ID
   c) Read account from VSAM ACCTFILE (ACCTDAT) for credit limit and status
   d) Read customer from VSAM CUSTFILE (CUSTDAT) for supplemental data
   e) IMS GU on PAUTSUM0 for the account (creates root if not found)
5 Apply business rules in sequence (fraud, account status, card status, credit limit)
6 Build CCPAURLY response with auth ID code, response code, and approved amount
7 MQPUT response to AWS.M2.CARDDEMO.PAUTH.REPLY with matching correlation ID
8 IMS REPL on PAUTSUM0: update counters, amounts, and credit balance
9 IMS ISRT new PAUTDTL1 child segment with full transaction detail; match status = 'P' (approved) or 'D' (declined)
10 Loop back to step 4 until MQRC-NO-MSG-AVAILABLE or 500-message limit reached
11 EXEC CICS DLI TERM (IMS PSB); close MQ queues (MQCLOSE); CICS RETURN

Summary View Flow (CPVS / COPAUS0C)

1 User initiates CPVS transaction from menu or directly from terminal
2 COPAUS0C reads COMMAREA for current account context (CDEMO-ACCT-ID)
3 Schedule IMS PSB; execute GU on PAUTSUM0 by account ID
4 Read VSAM ACCTFILE and CUSTFILE for supplemental display data
5 Display COPAU00 BMS map with summary statistics (counts, amounts, limits)
6 PF3 returns to previous screen; ENTER navigates to CPVD detail view

Detail View and Fraud Marking (CPVD / COPAUS1C + COPAUS2C)

1 User navigates to CPVD from CPVS (or directly)
2 COPAUS1C schedules IMS PSB; uses GNP to read PAUTDTL1 child segments
3 Display COPAU01 BMS map with transaction detail (merchant, amount, dates, match status)
4 Operator optionally sets fraud flag on screen
5 On ENTER with fraud flag set: COPAUS1C invokes COPAUS2C via EXEC CICS LINK
6 COPAUS2C executes DB2 INSERT into AUTHFRDS table with full authorization detail
7 COPAUS2C executes IMS REPL on PAUTDTL1: sets PA-AUTH-FRAUD='F', stores fraud report date
8 COPAUS1C displays updated screen; PF7/PF8 scrolls through additional detail records

Batch Jobs

CBPAUP0J — Expired Authorization Purge

* Program: CBPAUP0C.cbl  (IMS BMP batch program)
* JCL: jcl/CBPAUP0J.jcl
* PSB: PSBPAUTB (PCB offset +2 for batch mode)
* Parameter: PARM='30'  (P-EXPIRY-DAYS, 2-digit number of days)

Processing logic:

  1. Accept P-EXPIRY-DAYS from JCL PARM (e.g., '30' = 30-day threshold)
  2. Get current date in YYDDD (Julian) format via CICS ASKTIME / FORMATTIME equivalent
  3. IMS GN loop through all PAUTSUM0 root segments in the database
  4. For each root, IMS GNP through all PAUTDTL1 child segments
  5. For each child: compute Julian day difference between current date and PA-AUTH-DATE-9C
  6. If age > P-EXPIRY-DAYS AND match status = 'P' (pending):
    • Set match status to 'E' (expired)
    • Adjust PA-CREDIT-BALANCE in PAUTSUM0 to release held credit amount
    • IMS DLET the PAUTDTL1 segment; increment WS-NO-DTL-DELETED
  7. If all child segments deleted, IMS DLET the PAUTSUM0 root; increment WS-NO-SUMRY-DELETED
  8. Issue IMS checkpoint (WK-CHKPT-ID) periodically for warm restart support
  9. Print final statistics: total summary and detail records read/deleted; return code 0 on success

Acceptance Criteria Patterns

Authorization Processing (CP00): MQ message on REQUEST queue triggers CP00; program looks up card in XREFFILE, reads account from ACCTFILE; all 6 business rules evaluated; approved response has code '00' and approved amount = transaction amount; declined response has non-'00' code and approved amount = 0; IMS PAUTDTL1 inserted with match status 'P' (approved) or 'D' (declined); PAUTSUM0 updated with correct counters; response on REPLY queue within CICS timeout.
Batch Purge (CBPAUP0J): Job completes RC=0; all PAUTDTL1 records older than P-EXPIRY-DAYS with match status 'P' deleted from IMS; PA-CREDIT-BALANCE in PAUTSUM0 adjusted to release held credit; summary statistics log shows WS-NO-SUMRY-DELETED and WS-NO-DTL-DELETED counts; IMS checkpoints issued at configured intervals.
Fraud Marking (CPVD): Operator marks authorization as fraud on CPVD screen; DB2 AUTHFRDS row inserted with CARD_NUM, AUTH_TS, AUTH_FRAUD='F', and fraud report date; IMS PAUTDTL1 updated with PA-AUTH-FRAUD='F'; subsequent CP00 authorization request for same card declined with CARD-FRAUD reason flag.
