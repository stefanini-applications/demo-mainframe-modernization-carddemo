Optional Module IMS DB DB2 MQ CICS + Batch

📋 Module Overview

📦 Optional Module: Requires IMS DB, DB2, and MQ to be configured. Not part of base CardDemo installation. Located in app/app-authorization-ims-db2-mq/.

Purpose: Simulate credit card authorization request processing using MQ for async messaging, IMS DB for authorization record storage, and DB2 for audit logging. Demonstrates multi-system mainframe integration patterns.

Business Context: When a cardholder makes a purchase, a real-time authorization request is sent via MQ. The system processes the request, stores the pending authorization in IMS, and logs to DB2. Cardholders can view pending authorizations online before they post.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • View pending authorizations: As a cardholder, I want to view pending authorizations so I can see charges that are approved but not yet posted
  • Authorization detail: As a cardholder, I want to view authorization details (merchant, amount, timestamp) so I can verify recent activity
  • Process authorization: As an authorization system, I want to receive and process auth requests via MQ so authorizations are handled asynchronously without blocking cardholders
  • Purge expired auths: As a system operator, I want to run the purge job so expired pending authorizations are removed from IMS
  • MQ date inquiry: As a developer, I want to demonstrate MQ request/response by inquiring system date so the MQ integration pattern is validated

⚡ Development Acceleration Factors

  • MQ request/response pattern: CODATE01 and COACCT01 demonstrate the MQ trigger-and-reply pattern — reuse for new async flows
  • IMS DB access pattern: COPAUS0C/COPAUS1C show IMS segment read/update — template for new IMS-integrated features
  • DB2 audit pattern: COPAUS1C demonstrates DB2 INSERT for audit — reuse for any new audit logging requirement
  • Batch purge template: CBPAUP0C pattern for batch IMS record purge — reuse for other time-based cleanup jobs

📊 Complexity Guidelines

Simple (1–2 pts) Add display field to auth summary screen; change expiry threshold for purge job; add auth count display
Medium (3–5 pts) Add fraud flag field to authorization IMS segment; implement authorization approval/decline status update; add DB2 query for auth history
Complex (5–8 pts) Real-time fraud scoring integration via MQ; auto-decline rule engine; cross-channel auth consolidation (IMS + DB2 join reporting); auth-to-transaction reconciliation batch

🔧 Technical Foundation

Authorization Request Flow

External system sends auth request to MQ queue
  → MQ trigger fires CP00 (COPAUA0C)
  → COPAUA0C reads request from MQ
  → Validates card/account via VSAM XREFFILE
  → Inserts authorization record into IMS DB
  → Sends response back to MQ reply queue
  → Updates IMS record with response status

Pending Auth View Flow

Cardholder navigates to CPVS (COPAUS0C):
  → Reads IMS DB for pending auths by card number
  → Joins with VSAM for account/customer data
  → Displays summary list with amounts and merchants
  → User selects auth → navigates to CPVD (COPAUS1C)
  
CPVD (COPAUS1C):
  → Reads IMS segment for selected authorization
  → Displays full detail
  → On confirmation: updates IMS record status
  → Inserts DB2 audit row (auth ID, user, timestamp, action)

🔗 Dependencies

  • Base application: Must be fully deployed before this optional module can be installed
  • IMS DB: IMS database and segments must be defined and loaded before CPVS/CPVD/CP00 can run
  • DB2: CardDemo DB2 database must be created (CREADB21 JCL) before audit inserts
  • MQ: MQ queues must be configured and triggers defined for CP00 to receive messages
  • CBPAUP0J: Purge job must be scheduled separately from base batch sequence

✅ Acceptance Criteria Examples

Auth request processing:
Given a valid authorization request arrives on the MQ queue,
When CP00 (COPAUA0C) is triggered,
Then an IMS authorization record is created with status 'Pending'; a response is sent to the MQ reply queue with approved/declined status
Pending auth summary display:
Given pending authorizations exist in IMS for the current cardholder,
When navigating to CPVS,
Then authorization summary is displayed with merchant, amount, and timestamp
Authorization detail and DB2 audit:
Given a cardholder views authorization details on CPVD and confirms,
When pressing ENTER,
Then IMS record is updated with reviewed status; a DB2 audit row is inserted with user ID, auth ID, action, and current timestamp
Expired authorization purge:
Given authorizations older than the expiry threshold exist in IMS,
When CBPAUP0J runs,
Then expired authorization IMS segments are deleted; active/recent authorizations are retained; job completes with RC=0
MQ date inquiry (CDRD):
Given the MQ infrastructure is available,
When CDRD (CODATE01) transaction runs,
Then a date inquiry request is sent via MQ and the system date response is received and displayed
