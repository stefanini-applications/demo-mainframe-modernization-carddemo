1. Overview & Purpose

The accounts module is the central financial entity in the CardDemo mainframe application. It provides online (CICS) screens for viewing and updating credit card account details, and batch programs for account data processing, extract operations, and monthly interest calculation.

Every credit card, transaction, billing payment, and interest charge is ultimately linked to an account record in ACCTFILE. The account record stores the current balance, credit limit, cycle activity, and the ACCT-GROUP-ID that drives interest rate selection from the DISCGRP file.

Module ID: accounts

Layer: Business logic — CICS online + Batch

Primary VSAM files: ACCTFILE, XREFFILE, CUSTFILE, TCATBALF, DISCGRP

2. Architectural Context

CardDemo uses a classic mainframe OLTP architecture: CICS programs handle online 3270 terminal interactions via BMS maps, while batch COBOL programs run under JCL on z/OS. Both tiers share the same VSAM KSDS files.

Module Position in System

  [Authentication (CC00)]
         │  COMMAREA (CDEMO-ACCT-ID / CDEMO-CARD-NUM)
         ▼
  [Main Menu (CM00 / COMEN01C)]
         │
         ├──► [Account View   CAVW / COACTVWC]  ──► reads XREFFILE → ACCTFILE → CUSTFILE
         │           │ PF5 or ENTER
         └──► [Account Update CAUP / COACTUPC]  ──► reads + rewrites ACCTFILE + CUSTFILE
                                                     (with optimistic locking)

  [Batch — INTCALC job]
     CBACT04C reads TCATBALF (sequential)
         → XREFFILE (AIX by ACCT-ID)
         → ACCTFILE (random)
         → DISCGRP  (random)
         → computes monthly interest
         → writes interest transaction to TRANSACT output file
Key architectural pattern: The accounts module uses a read-cross-reference → read-master → read-customer chain. Card number (from COMMAREA) is the entry point; Account ID and Customer ID are resolved through XREFFILE.

5. Data Models

ACCOUNT-RECORD — ACCTFILE (CVACT01Y.cpy)

01  ACCOUNT-RECORD.
    05  ACCT-ID                   PIC 9(11).        *> Primary key (11-digit)
    05  ACCT-ACTIVE-STATUS        PIC X(01).        *> 'Y'=Active, 'N'=Inactive
    05  ACCT-CURR-BAL             PIC S9(10)V99.    *> Current balance (signed)
    05  ACCT-CREDIT-LIMIT         PIC S9(10)V99.    *> Total credit limit
    05  ACCT-CASH-CREDIT-LIMIT    PIC S9(10)V99.    *> Cash advance limit
    05  ACCT-OPEN-DATE            PIC X(10).        *> YYYY-MM-DD
    05  ACCT-EXPIRAION-DATE       PIC X(10).        *> YYYY-MM-DD (note: field name typo in source)
    05  ACCT-REISSUE-DATE         PIC X(10).        *> YYYY-MM-DD
    05  ACCT-CURR-CYC-CREDIT      PIC S9(10)V99.    *> Cycle credits (payments received)
    05  ACCT-CURR-CYC-DEBIT       PIC S9(10)V99.    *> Cycle debits (charges made)
    05  ACCT-ADDR-ZIP             PIC X(10).        *> Billing ZIP code
    05  ACCT-GROUP-ID             PIC X(10).        *> Links to DISCGRP interest rates
    05  FILLER                    PIC X(178).       *> Total record = 300 bytes

CARD-XREF-RECORD — XREFFILE (CVACT03Y.cpy)

01  CARD-XREF-RECORD.
    05  XREF-CARD-NUM             PIC X(16).        *> Primary key (card number)
    05  XREF-CUST-ID              PIC 9(09).        *> Customer ID
    05  XREF-ACCT-ID              PIC 9(11).        *> Account ID
    05  FILLER                    PIC X(14).        *> Total record = 50 bytes

*> Alternate Index: XREF-ACCT-ID (used in CBACT04C to look up card by account)

CARD-RECORD — CARDFILE (CVACT02Y.cpy)

01  CARD-RECORD.
    05  CARD-NUM                  PIC X(16).        *> Primary key (16-digit PAN)
    05  CARD-ACCT-ID              PIC 9(11).        *> Foreign key → ACCTFILE
    05  CARD-CVV-CD               PIC 9(03).        *> CVV security code
    05  CARD-EMBOSSED-NAME        PIC X(50).        *> Name on card
    05  CARD-EXPIRAION-DATE       PIC X(10).        *> YYYY-MM-DD
    05  CARD-ACTIVE-STATUS        PIC X(01).        *> 'Y'=Active, 'N'=Inactive
    05  FILLER                    PIC X(59).        *> Total record = 150 bytes

TRAN-CAT-BAL-RECORD — TCATBALF (CVTRA01Y.cpy)

*> Input to CBACT04C interest calculation — keyed by account+type+category
01  TRAN-CAT-BAL-RECORD.
    05  TRAN-CAT-KEY.
        10  TRANCAT-ACCT-ID       PIC 9(11).        *> Account ID
        10  TRANCAT-TYPE-CD       PIC X(02).        *> Transaction type code
        10  TRANCAT-CD            PIC 9(04).        *> Transaction category code
    05  TRAN-CAT-BAL              PIC S9(09)V99.    *> Category balance (basis for interest)

DIS-GROUP-RECORD — DISCGRP (CVTRA02Y.cpy)

*> Interest rate lookup — keyed by account group + transaction type + category
01  DIS-GROUP-RECORD.
    05  DIS-GROUP-KEY.
        10  DIS-ACCT-GROUP-ID     PIC X(10).        *> Maps from ACCT-GROUP-ID
        10  DIS-TRAN-TYPE-CD      PIC X(02).
        10  DIS-TRAN-CAT-CD       PIC 9(04).
    05  DIS-INT-RATE              PIC S9(09)V99.    *> Annual interest rate (%)
    05  DIS-GROUP-DESC            PIC X(20).

*> DISCGRP key = ACCT-GROUP-ID + TRAN-TYPE-CD + TRAN-CAT-CD
*> Default key uses 'DEFAULT   ' as group ID if specific group not found

6. Processing Flows

Account View Flow (COACTVWC — CAVW)

1. CICS transaction CAVW invoked; COMMAREA contains CDEMO-CARD-NUM
2. Program COACTVWC receives control

   IF first entry (EIBCALEN = 0):
     → Prompt user for Account ID input on screen

   IF re-entry (EIBCALEN > 0):
     → Read input from BMS map (CACTVWAI)
     → Validate Account ID field (non-blank, numeric)

     IF valid Account ID entered:
       9200: READ XREFFILE (AIX path by ACCT-ID)
             → Sets XREF-CUST-ID → CDEMO-CUST-ID
             → Sets XREF-CARD-NUM → CDEMO-CARD-NUM

       9300: READ ACCTFILE (primary key = ACCT-ID)
             → Loads ACCOUNT-RECORD into working storage

       9400: READ CUSTFILE (primary key = CUST-ID)
             → Loads CUSTOMER-RECORD into working storage

     1200: Map account fields to screen output area (CACTVWAO):
           - ACCT-ACTIVE-STATUS, ACCT-CURR-BAL, ACCT-CREDIT-LIMIT
           - ACCT-CASH-CREDIT-LIMIT, ACCT-CURR-CYC-CREDIT/DEBIT
           - ACCT-OPEN-DATE, ACCT-EXPIRAION-DATE, ACCT-REISSUE-DATE
           - ACCT-GROUP-ID, ACCT-ADDR-ZIP
           - CUST-FIRST/MIDDLE/LAST-NAME, CUST-SSN (formatted NNN-NN-NNNN)
           - CUST-DOB-YYYY-MM-DD, CUST-FICO-CREDIT-SCORE
           - CUST-ADDR-LINE-1/2/3, CUST-ADDR-STATE-CD, CUST-ADDR-ZIP
           - CUST-PHONE-NUM-1/2, CUST-GOVT-ISSUED-ID, CUST-EFT-ACCOUNT-ID

     1400: EXEC CICS SEND MAP(COACTVW) FROM(CACTVWAO) ERASE FREEKB

3. EXEC CICS RETURN TRANSID(CAVW) COMMAREA(WS-COMMAREA)

Account Update Flow (COACTUPC — CAUP)

1. CICS transaction CAUP invoked from Account View (PF5)
2. Program COACTUPC receives control with pre-populated COMMAREA

   FIRST PASS — Populate screen with current data:
     → Read XREFFILE → ACCTFILE → CUSTFILE (same chain as CAVW)
     → Display all editable fields on CACTUPA map
     → Store "old values" in screen hidden fields (ACUP-OLD-*)
     → EXEC CICS RETURN TRANSID(CAUP) COMMAREA

   SECOND PASS — Process user edits:
     → Receive map input (CACTUAI)
     → Validate all changed fields:
         - Active status: must be 'Y' or 'N'
         - Numeric fields: credit limit, balance, cycle amounts
         - Dates: YYYY-MM-DD format, open/expiry/reissue
         - ZIP code: 5 or 9 digit alphanumeric
         - Phone numbers: (NNN)NNN-NNNN format
         - SSN: NNN-NN-NNNN format
         - FICO score: numeric 0–850

     IF all edits pass:
       9600-WRITE-PROCESSING:
         READ ACCTFILE UPDATE (CICS record-level lock)
         READ CUSTFILE UPDATE (CICS record-level lock)

         9700-CHECK-CHANGE-IN-REC:
           Compare all ACUP-OLD-* fields against freshly-read record
           IF data changed between view and update → abort, display error

         Prepare ACCT-UPDATE-RECORD from ACUP-NEW-* fields
         Prepare CUST-UPDATE-RECORD from ACUP-NEW-CUST-* fields

         EXEC CICS REWRITE FILE(ACCTFILE) FROM(ACCT-UPDATE-RECORD)
         EXEC CICS REWRITE FILE(CUSTFILE) FROM(CUST-UPDATE-RECORD)

         IF CUSTFILE rewrite fails:
           EXEC CICS SYNCPOINT ROLLBACK   ← rolls back both file updates

3. Display success or error message; RETURN TRANSID(CAUP) COMMAREA

Interest Calculation Batch Flow (CBACT04C — INTCALC)

Program: CBACT04C
Files:  TCATBALF (input, sequential)
        XREFFILE  (input, random, AIX by ACCT-ID)
        ACCTFILE  (input/output, random)
        DISCGRP   (input, random)
        TRANSACT  (output, sequential — interest transaction records)

MAINLINE:
  Open all files
  Read TCATBALF sequentially until EOF

    FOR EACH TRAN-CAT-BAL-RECORD:
      1100: READ XREFFILE AIX (by FD-XREF-ACCT-ID = TRANCAT-ACCT-ID)
            → get XREF-CARD-NUM

      1100: READ ACCTFILE random (by ACCT-ID)
            → get ACCT-CURR-BAL and ACCT-GROUP-ID

      1200: Build DISCGRP key = ACCT-GROUP-ID + TRAN-TYPE-CD + TRAN-CAT-CD
            READ DISCGRP random
            IF key not found (status '23'):
              Retry with key = 'DEFAULT   ' + TYPE + CAT

      1300: COMPUTE WS-MONTHLY-INT = (TRAN-CAT-BAL * DIS-INT-RATE) / 1200
            ADD WS-MONTHLY-INT TO WS-TOTAL-INT

      1300-B: Build TRAN-RECORD for interest transaction:
              TRAN-ID         = PARM-DATE + suffix counter
              TRAN-TYPE-CD    = '01'
              TRAN-CAT-CD     = '05' (hard-coded)
              TRAN-SOURCE     = 'System'
              TRAN-DESC       = 'Int. for a/c ' + ACCT-ID
              TRAN-AMT        = WS-MONTHLY-INT
              TRAN-CARD-NUM   = XREF-CARD-NUM
              TRAN-ORIG/PROC-TS = current timestamp
              WRITE FD-TRANFILE-REC

  Close all files
  Display total interest computed

7. Business Rules

Account Status

  • ACCT-ACTIVE-STATUS = 'Y' is required for transactions and billing payments to proceed
  • Status can be toggled via Account Update (CAUP) — valid values are 'Y' and 'N' only
  • Inactive accounts ('N') are visible in Account View but downstream modules (billing, transactions) check this flag before proceeding

Account Lookup Strategy

  • Primary entry: Card Number in COMMAREA → XREFFILE AIX (by card number) → resolves ACCT-ID and CUST-ID
  • Direct entry: Account ID entered on Account View screen → XREFFILE AIX (by ACCT-ID) → resolves CARD-NUM and CUST-ID
  • Both lookup paths populate COMMAREA fields for downstream use

Optimistic Locking (COACTUPC)

  • When Account View is displayed, old field values are stored in ACUP-OLD-* hidden fields
  • On update submit, program compares old values against freshly-read VSAM records
  • If any field differs between old and current VSAM data, update is rejected with "data changed" error — user must re-enter
  • CICS record-level locking via READ … UPDATE prevents concurrent overwrites
  • If REWRITE CUSTFILE fails after REWRITE ACCTFILE succeeded, EXEC CICS SYNCPOINT ROLLBACK rolls back the account rewrite

Interest Calculation Rules

  • Monthly interest = (TRAN-CAT-BAL × DIS-INT-RATE) / 1200
  • Rate is an annual percentage; dividing by 1200 converts to monthly decimal
  • Rate lookup key: ACCT-GROUP-ID + TRAN-TYPE-CD + TRAN-CAT-CD
  • If specific group+type+category record not found in DISCGRP, falls back to DEFAULT group key
  • Each category balance generates a separate interest transaction record (type 01, category 05)
  • Fee calculation (1400-COMPUTE-FEES) is present in code but not yet implemented ("to be implemented")

Credit Limit Enforcement

  • Credit limit is stored in ACCTFILE and enforced by the transactions module (COTRN02C) during transaction add
  • The accounts module itself allows the credit limit field to be updated directly via CAUP
  • Both ACCT-CREDIT-LIMIT (total) and ACCT-CASH-CREDIT-LIMIT (cash advance) are maintained separately

Date Formatting

  • All dates stored in ACCTFILE use PIC X(10) format YYYY-MM-DD
  • COACTUPC builds dates by concatenating separate year/month/day screen fields with '-' delimiters via STRING
  • COACTVWC displays dates directly from ACCTFILE (no reformatting)

Purpose: Compute monthly interest for each account/transaction-type/category combination and write interest transaction records.

11. Error Handling

CICS File Error Handling (COACTVWC / COACTUPC)

EVALUATE WS-RESP-CD
    WHEN DFHRESP(NORMAL)
        CONTINUE
    WHEN DFHRESP(NOTFND)
        SET INPUT-ERROR TO TRUE
        SET FLG-ACCTFILTER-NOT-OK TO TRUE
        STRING 'Account:' WS-CARD-RID-ACCT-ID-X
               ' not found in Acct Master file. Resp:'
               ERROR-RESP ' Reas:' ERROR-RESP2
               DELIMITED BY SIZE INTO WS-RETURN-MSG
    WHEN OTHER
        MOVE 'READ'       TO ERROR-OPNAME
        MOVE LIT-ACCTFILENAME TO ERROR-FILE
        MOVE WS-RESP-CD   TO ERROR-RESP
        MOVE WS-REAS-CD   TO ERROR-RESP2
        MOVE WS-FILE-ERROR-MESSAGE TO WS-RETURN-MSG
END-EVALUATE

12. Development Guidelines & Patterns

Adding a New Display Field to Account View

  1. Add field definition to COACTVW.bms (new DFHMDF entry in CACTVWA map)
  2. Re-assemble BMS map to regenerate COACTVW.CPY
  3. In COACTVWC.cbl paragraph 1200-SETUP-SCREEN-VARS: add MOVE statement mapping source field to new output field
  4. Recompile COACTVWC.cbl
  5. No VSAM changes required if field already exists in ACCTFILE or CUSTFILE

Adding a New Updatable Account Field

  1. Add field to COACTUP.bms (input + output variants)
  2. Add old-value shadow field (ACUP-OLD-newfield) to update working storage for optimistic locking
  3. Add validation paragraph in COACTUPC.cbl following existing WS-EDIT-* pattern
  4. Add comparison in 9700-CHECK-CHANGE-IN-REC paragraph
  5. Add MOVE from ACUP-NEW-newfield to ACCT-UPDATE-newfield in 9600-WRITE-PROCESSING
  6. If field is new to ACCTFILE, update CVACT01Y.cpy and re-define VSAM RECLN (breaking change)

Adding a New Interest Rate Tier

  1. Add new DISCGRP record with a new DIS-ACCT-GROUP-ID + type + category combination
  2. Set ACCT-GROUP-ID on affected accounts via COACTUPC or batch update
  3. No code change to CBACT04C required if rate is in DISCGRP (lookup is data-driven)
  4. If a new transaction category (TRAN-CAT-CD) is needed, update TCATBALF loading JCL

Code Patterns to Follow

*  Pattern: CICS file read with full error handling
   EXEC CICS READ
        DATASET   (LIT-ACCTFILENAME)
        RIDFLD    (WS-CARD-RID-ACCT-ID-X)

Known Tech Debt & Gotchas

  • Typo in field name: ACCT-EXPIRAION-DATE (missing 'T') is present in CVACT01Y.cpy and all programs — do not "fix" without updating all copybook references simultaneously
  • Fee calculation stub: 1400-COMPUTE-FEES in CBACT04C exits immediately ("to be implemented") — not a bug, but a known gap
  • SSN display: COACTVWC masks SSN display using STRING with dashes but stores it without formatting in CUSTFILE — consistent with PII handling in demo context
  • Fixed RECLN=300: Any addition of fields to ACCOUNT-RECORD requires reducing FILLER (currently 178 bytes) and redefining the VSAM cluster
  • CICS file close/open: Batch programs MUST be preceded by CLOSEFIL JCL — skipping this risks data corruption or VSAM OPEN errors

13. User Story Templates

View Account Stories

  • As a cardholder, I want to view my current account balance so I know how much I owe
  • As a cardholder, I want to see my credit limit and available credit so I can plan purchases
  • As a cardholder, I want to view my account's expiration date so I know when to expect a renewal card
  • As a cardholder, I want to see current cycle credits and debits so I can track my monthly activity
  • As a cardholder, I want to view my associated customer information (address, phone) so I can confirm it is up to date

Update Account Stories

  • As a cardholder, I want to update my billing ZIP code so my address on file is accurate
  • As an administrator, I want to change an account's active status to 'N' so unauthorized use is prevented
  • As an administrator, I want to update the credit limit so an account reflects an approved limit change
  • As an administrator, I want to update customer contact information so records stay current
  • As a cardholder, I want to update my address so my statements are mailed correctly

Batch / System Stories

  • As a system, I want to calculate monthly interest for each account/type/category so balances reflect accrued charges at cycle end
  • As a system, I want to apply the DEFAULT interest rate when a specific group rate is missing so calculation never fails silently
  • As a system operator, I want to extract account data to a flat file so downstream reporting jobs can process it
  • As a system operator, I want to validate account records in batch so data integrity issues are caught before online use

14. Acceptance Criteria Examples

AC-ACC-001 — Account view success:
Given a signed-in cardholder with a valid Account ID,
When the cardholder navigates to Account View (CAVW) and enters their Account ID,
Then the screen displays: current balance, credit limit, cash credit limit, active status, open date, expiration date, reissue date, group ID, and associated customer name/address/FICO score — all sourced from ACCTFILE and CUSTFILE via XREFFILE.
AC-ACC-002 — Account ID not found:
Given a cardholder enters an Account ID that does not exist in XREFFILE,
When ENTER is pressed,
Then the Account ID field is highlighted RED, an error message is displayed, and no account data appears on screen. No VSAM update occurs.
AC-ACC-003 — Account update — ZIP code:
Given a signed-in cardholder on Account Update (CAUP) with a valid account displayed,
When the cardholder changes the ZIP code field and presses ENTER,
Then ACCT-ADDR-ZIP in ACCTFILE is rewritten with the new value, a success message is shown, and the screen redisplays with the updated ZIP.
AC-ACC-004 — Optimistic locking conflict:
Given two users simultaneously view the same account,
When the first user updates and saves, and then the second user attempts to save,
Then the second user receives a "data changed" error message, no VSAM rewrite occurs, and the second user must re-enter their changes based on the current data.
AC-ACC-005 — Concurrent update rollback:
Given a user submits account and customer updates,
When REWRITE ACCTFILE succeeds but REWRITE CUSTFILE fails,
Then EXEC CICS SYNCPOINT ROLLBACK is issued, both updates are rolled back atomically, and an error message is displayed to the user.
AC-ACC-006 — Interest calculation (INTCALC):
Given TCATBALF contains category balance records and DISCGRP has matching rate entries,
When CBACT04C completes with return code 0,
Then for each TRAN-CAT-BAL-RECORD, an interest transaction is written to TRANSACT with amount = (TRAN-CAT-BAL × DIS-INT-RATE) / 1200, and the program displays the total interest computed to SYSOUT.
AC-ACC-007 — DEFAULT interest rate fallback:
Given an account has a group ID that does not exist in DISCGRP for a given type+category,
When CBACT04C processes that category balance,
Then the program automatically retries with group key 'DEFAULT ' and applies that rate. No ABEND occurs.
AC-ACC-008 — Inactive account displayed:
Given an account with ACCT-ACTIVE-STATUS = 'N',
When the cardholder views the account (CAVW),
Then ACSTTUSO displays 'N', the account record is shown in full, and downstream transaction/billing modules reject operations for this account.

15. Complexity Guidelines

Simple (1–2 pts)
  • Add new read-only display field to account view (field already in ACCTFILE or CUSTFILE)
  • Change field label or color on BMS map
  • Add new display-only field to update screen
  • Add a new DISCGRP rate entry (data change only, no code change)
Medium (3–5 pts)
  • Add new updatable account field with full validation + optimistic lock check
  • Modify interest formula for a new rate tier in CBACT04C
  • Add new customer field to update screen (validation + REWRITE)
  • Add account lookup by customer SSN (new XREFFILE AIX or join)
  • Implement the fee calculation stub (1400-COMPUTE-FEES)
Complex (5–8 pts)
  • Add new field to ACCOUNT-RECORD (VSAM RECLN change → file redefinition + all batch program updates + data migration)
  • Add account history/audit trail (new VSAM file, new copybook, new batch program)
  • New interest tier requiring DISCGRP schema change (new key structure)
  • Account statement integration (CREASTMT) — reads ACCTFILE + TRANFILE + CUSTFILE sequentially
  • MQ-based account inquiry integration (authorization module pattern)
