User Story Development Guide — CardDemo Mainframe Application

CICS COBOL VSAM ACCTFILE + TRANFILE Transaction: CB00

📋 Module Overview

Purpose: Process bill payments for a credit card account. Accepts a payment amount from the cardholder, creates a credit transaction record, and reduces the account's current balance.

Business Context: Payment processing is the primary cardholder action to reduce outstanding balance. Payments are recorded as positive (credit) transactions and immediately update the account balance. The billing flow depends on accounts and transactions modules.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Make payment: As a cardholder, I want to make a bill payment so my outstanding balance is reduced
  • View balance before paying: As a cardholder, I want to see my current balance on the payment screen so I know how much to pay
  • Partial payment: As a cardholder, I want to enter a partial payment amount so I can pay more than the minimum but less than the full balance
  • Payment confirmation: As a cardholder, I want to receive a confirmation message after submitting a payment so I know it was processed
  • Payment history: As a cardholder, I want to see my payment transactions in the transaction list so I have a record of all payments made

⚡ Development Acceleration Factors

  • TRAN-RECORD structure: Payment creates a standard TRAN-RECORD with positive TRAN-AMT — no new data structures needed
  • Account balance update: Same ACCTFILE REWRITE pattern as POSTTRAN batch — well-established approach
  • Single-screen flow: COBIL00C is self-contained — input, validation, and update in one CICS program
  • Existing validation patterns: Amount format validation reuses patterns from Transaction Add (COTRN02C)

📊 Complexity Guidelines

Simple (1–2 pts) Add minimum payment display; show last payment date on screen; add payment amount field validation message improvement
Medium (3–5 pts) Add minimum payment calculation and enforcement; implement payment due date display; add statement balance vs. current balance distinction
Complex (5–8 pts) EFT/ACH payment integration; scheduled auto-payment batch job; payment reversal capability; late fee calculation and application

🔧 Technical Foundation

Bill Payment Flow

Entry: ACCT-ID + CARD-NUM in COMMAREA
  → Display COBIL00 screen with current balance from ACCTFILE
  → User enters payment amount → ENTER
  → Validate: amount > 0, numeric, not exceeding balance rules
  → Read ACCTFILE (key=ACCT-ID) for current balance
  → Generate payment TRAN-ID
  → WRITE TRANFILE:
      TRAN-TYPE-CD  = payment type code
      TRAN-AMT      = +positive payment amount
      TRAN-CARD-NUM = current card
      TRAN-ORIG-TS  = current timestamp
  → Update ACCTFILE:
      ACCT-CURR-BAL        -= payment amount
      ACCT-CURR-CYC-CREDIT += payment amount
  → Display confirmation message

🔗 Dependencies

  • accounts: ACCT-ID must be set in COMMAREA; account must be ACTIVE (ACCT-ACTIVE-STATUS='Y')
  • transactions: Payment creates a TRAN-RECORD in TRANFILE — visible in transaction list as a credit entry
  • ACCTFILE: Balance update is a direct REWRITE — subject to CLOSEFIL dependency if batch is also running
  • TRANFILE: Payment transaction written before balance update; if TRANFILE write fails, balance update does not proceed

Business Rules

  • Payment amount must be greater than zero
  • No minimum payment enforcement in base application (demo simplification)
  • Account must be active (ACCT-ACTIVE-STATUS = 'Y') to accept payment
  • Payment creates a positive (credit) TRAN-RECORD entry — visible in transaction history
  • ACCT-CURR-CYC-CREDIT is incremented by payment amount for cycle tracking

✅ Acceptance Criteria Examples

Successful payment:
Given a cardholder with an active account and balance of $500,
When entering a payment of $100 and pressing ENTER,
Then ACCT-CURR-BAL is reduced to $400; a credit TRAN-RECORD is written; confirmation message displayed
Invalid payment amount — zero:
Given a payment amount of $0 or blank,
When pressing ENTER,
Then the screen redisplays with "Amount must be greater than zero" and no VSAM updates occur
Invalid payment amount — non-numeric:
Given a non-numeric entry in the payment field,
When pressing ENTER,
Then field validation error is displayed and cursor repositioned on the amount field
Payment on inactive account:
Given an account with ACCT-ACTIVE-STATUS = 'N',
When attempting a payment,
Then the system displays "Account is not active" and no payment is processed
Payment appears in transaction list:
Given a payment was successfully processed,
When navigating to Transaction List (CT00),
Then the payment entry appears with positive amount and payment type code
