2. Architectural Context

The billing module follows the standard CardDemo CICS architecture pattern:

  • One transaction, one program, one BMS mapset — CICS transaction CB00 invokes COBIL00C which uses mapset COBIL00.
  • COMMAREA-driven state — Session context (user ID, account ID, card number) flows in via CARDDEMO-COMMAREA (copybook COCOM01Y). The billing program also stores per-transaction state in the CDEMO-CB00-INFO group.
  • Pseudo-conversational CICS — Each ENTER/PF key press causes the program to RETURN TRANSID(CB00) COMMAREA(...), preserving state across terminal interactions without holding server resources.
  • VSAM direct access — Account record is read with UPDATE intent for the subsequent REWRITE; no intermediate batch coordination is required for online payment.

Module Position in Navigation Flow

Authentication (CC00)
  └─→ Main Menu (CM00/COMEN01C)
        └─→ Bill Payment (CB00/COBIL00C)   ← THIS MODULE
 reads  ACCTFILE   (account balance)
              ├─→ reads  XREFFILE   (card-account cross-ref, via CXACAIX AIX)
              ├─→ writes TRANFILE   (new payment transaction record)
              └─→ rewrites ACCTFILE (updated balance = 0)

7. Data Models & VSAM Files

Account Record (ACCTFILE / ACCTDAT)

Key fields used by the billing module (from CVACT01Y.cpy):

01  ACCOUNT-RECORD.
    05  ACCT-ID                  PIC 9(11)       -- Primary key; entered by user as ACTIDIN
    05  ACCT-ACTIVE-STATUS       PIC X(01)       -- Must be 'Y' (Active) to process payment
    05  ACCT-CURR-BAL            PIC S9(10)V99   -- Current balance; DISPLAYED and used as payment amount
    05  ACCT-CURR-CYC-CREDIT     PIC S9(10)V99   -- Incremented by payment amount (not currently in billing code)
    05  ACCT-CREDIT-LIMIT        PIC S9(10)V99   -- Not checked in billing; for reference only
    05  ACCT-GROUP-ID            PIC X(10)       -- Interest group; not used in billing
Note on ACCT-CURR-CYC-CREDIT: The system overview references this field being updated, but inspection of COBIL00C.cbl shows only ACCT-CURR-BAL is modified (ACCT-CURR-BAL = ACCT-CURR-BAL - TRAN-AMT). ACCT-CURR-CYC-CREDIT is NOT updated by the current billing program. This may be a documentation discrepancy or an intentional simplification in the demo.

Card-Account Cross-Reference (XREFFILE / CXACAIX AIX)

Key fields used (from CVACT03Y.cpy):

01  CARD-XREF-RECORD.
    05  XREF-CARD-NUM   PIC X(16)    -- Returned value: primary card number
    05  XREF-CUST-ID    PIC 9(09)    -- Not used by billing
    05  XREF-ACCT-ID    PIC 9(11)    -- AIX key: searched by account ID to get card number

    05  TRAN-ID              PIC X(16)       -- System-generated: max existing + 1

8. Business Rules

BR-BIL-001 — Account ID is required
The ACTIDIN field must not be blank or low-values. Error: "Acct ID can NOT be empty..."
BR-BIL-002 — Account must exist
ACCTFILE READ must return DFHRESP(NORMAL). If NOTFND: "Account ID NOT found..." Error. No payment proceeds.
BR-BIL-003 — Balance must be positive
ACCT-CURR-BAL must be greater than zero. If zero or negative: "You have nothing to pay..." No payment proceeds.
BR-BIL-004 — Confirmation required
After balance is displayed, the user must enter 'Y' (or 'y') in the CONFIRM field and press ENTER to execute the payment. Any other non-Y input (blank, N, n, other) does not process the payment.
BR-BIL-005 — Payment is always full balance
The payment amount is always ACCT-CURR-BAL — the entire current outstanding balance. There is no partial payment entry. After payment, ACCT-CURR-BAL is set to zero.
BR-BIL-006 — Transaction record created for every payment
A TRAN-RECORD is written to TRANFILE with type code '02', category 2, and the full balance amount. This creates an audit trail and is visible in the Transaction List (CT00).
BR-BIL-007 — Transaction ID is sequential
New TRAN-ID = (highest existing TRAN-ID in TRANFILE) + 1. This is determined by a backwards browse of TRANFILE. Risk: concurrent writes could produce duplicate IDs.
BR-BIL-008 — Card number resolved via XREFFILE
The card number is NOT entered by the user. It is resolved from the XREFFILE alternate index (CXACAIX) using the account ID. The first (lowest) card number linked to the account is used.
BR-BIL-009 — No minimum payment enforcement
There is no minimum payment amount check. The demo application always pays in full. Minimum payment logic is a known gap for modernisation.
BR-BIL-010 — No ACCT-CURR-CYC-CREDIT update
Despite documentation stating that ACCT-CURR-CYC-CREDIT is incremented, the source code does not perform this update. Only ACCT-CURR-BAL is modified.

Confirmation Field Behaviour

13. User Story Templates

Core Stories

  • As a cardholder, I want to view my current balance on the bill payment screen so I know my total outstanding amount before confirming payment
  • As a cardholder, I want to pay my outstanding balance in full so my account is cleared and no interest accrues on the paid amount
  • As a cardholder, I want to receive a transaction ID after a successful payment so I have an audit reference for the payment
  • As a cardholder, I want to see a confirmation prompt before my payment is processed so I can avoid accidental payments
  • As a cardholder, I want to cancel a payment using PF3 so I can return to the main menu without making a payment

Enhancement Stories (Modernisation Candidates)

  • As a cardholder, I want to enter a partial payment amount so I can pay more than the minimum but less than the full balance
  • As a cardholder, I want to see the minimum payment due on the payment screen so I know the lowest amount I must pay
  • As a cardholder, I want to see my payment due date on the screen so I can avoid late fees
  • As a cardholder, I want to view my payment history within the billing screen so I can confirm past payments without navigating to Transaction List
  • As a system, I want to update ACCT-CURR-CYC-CREDIT when a payment is processed so cycle-end reporting is accurate
  • As a system operator, I want to run a scheduled auto-payment batch job so eligible accounts with auto-pay enabled are paid on the due date without manual cardholder action
  • As a cardholder, I want to reverse a mistaken payment so my balance is restored when an incorrect payment is submitted
  • As a cardholder, I want to pay via EFT/ACH from my bank account so electronic funds are transferred directly

User Story Pattern

Pattern: As a [cardholder / system / system operator],
         I want to [payment action],
         so that [financial outcome / compliance / usability goal]

Batch Interaction Notes

The billing module accesses ACCTFILE and TRANFILE online. If batch jobs (POSTTRAN, INTCALC) are running simultaneously, CICS files must be managed carefully:
  • CLOSEFIL job closes CICS datasets before batch runs that update them
  • OPENFIL job reopens them after batch completion
  • Online billing cannot run while ACCTDAT or TRANSACT are closed
