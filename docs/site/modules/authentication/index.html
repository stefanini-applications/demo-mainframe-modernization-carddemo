1. Overview and Purpose

The Authentication module is the entry point for all CardDemo application interactions. It implements user signon and session initialization for the CardDemo mainframe credit card management application running under IBM CICS (Customer Information Control System) on z/OS.

Every user, whether a regular cardholder or a system administrator, must authenticate through this module before accessing any other function. Upon successful authentication, the module initializes the shared session state (COMMAREA) that all downstream CICS programs rely upon for navigation and access control.

Architectural Significance: Authentication is a prerequisite for all other modules. No end-to-end test of any feature can succeed without authentication working correctly. The CDEMO-USER-TYPE value set here controls role-based navigation throughout the entire application.

Module Responsibilities

  • Display the 3270 signon screen (COSGN00) to the terminal operator
  • Accept User ID (up to 8 characters) and Password (up to 8 characters) input
  • Validate credentials by performing a keyed read against the USRSEC VSAM KSDS file
  • Initialize CARDDEMO-COMMAREA with the authenticated user identity and role
  • Route authenticated users to the appropriate menu: Main Menu (regular) or Admin Menu (admin)
  • Display contextual error messages for invalid credentials, missing fields, or unexpected keys
  • Provide a clean exit path via PF3 with a thank-you message

2. Architectural Context

CardDemo is a mainframe OLTP application using CICS for online transaction processing and VSAM for persistent data storage. The authentication module follows the standard CardDemo screen/program pattern: a BMS map defines the 3270 terminal screen layout, and a COBOL program contains all processing logic. Communication between programs uses the CICS COMMAREA mechanism.

Online Layer (CICS)
  • Transaction ID: CC00
  • Program: COSGN00C
  • Screen mapset: COSGN00, map: COSGN0A
  • Interface: IBM 3270 terminal (no REST/HTTP)
  • Model: Pseudo-conversational
Data Layer (VSAM)
  • File: USRSEC (KSDS, RECLN=80)
  • Primary key: SEC-USR-ID PIC X(08)
  • Access method: Direct keyed READ only
  • No Alternate Index required
  • Pre-loaded via DUSRSECJ JCL job

Session State Propagation

Authentication is the only module that creates the session. All other modules consume the COMMAREA values set here. The COMMAREA is passed via EXEC CICS XCTL COMMAREA and EXEC CICS RETURN COMMAREA, maintaining session continuity across all subsequent transactions. 

CARDDEMO-COMMAREA fields populated by authentication:
  CDEMO-FROM-TRANID  = 'CC00'      (source transaction)
  CDEMO-FROM-PROGRAM = 'COSGN00C'  (source program)
  CDEMO-USER-ID      = entered User ID (uppercased)
  CDEMO-USER-TYPE    = 'A' or 'U'  (drives all menu routing)
  CDEMO-PGM-CONTEXT  = 0           (fresh entry context)

6. Data Models

SEC-USER-DATA - USRSEC VSAM Record (CSUSR01Y.cpy)

* File: app/cpy/CSUSR01Y.cpy
* VSAM: USRSEC (KSDS, RECLN=80)
* Primary key: SEC-USR-ID (PIC X(08), offset 0, length 8)

 01 SEC-USER-DATA.
   05 SEC-USR-ID      PIC X(08).   *> User ID (primary key)
   05 SEC-USR-FNAME   PIC X(20).   *> First name
   05 SEC-USR-LNAME   PIC X(20).   *> Last name
   05 SEC-USR-PWD     PIC X(08).   *> Password (plain text - demo only)
   05 SEC-USR-TYPE    PIC X(01).   *> 'A'=Admin, 'U'=Regular User
   05 SEC-USR-FILLER  PIC X(23).   *> Reserved (total = 80 bytes)
Security Note: Passwords are stored in plain text in the USRSEC file. This is an intentional characteristic of the CardDemo demo. Production implementations must replace this with a cryptographic hash scheme (e.g., SHA-256 with salt).

8. Processing Flow

Cold Start (EIBCALEN = 0)

1. CICS initiates CC00 transaction with no COMMAREA (EIBCALEN=0)
2. Program moves LOW-VALUES to COSGN0AO (clears output map)
3. Sets cursor to USERID field: MOVE -1 TO USERIDL OF COSGN0AI
4. Calls SEND-SIGNON-SCREEN (EXEC CICS SEND MAP ERASE CURSOR)
5. EXEC CICS RETURN TRANSID('CC00') COMMAREA(...) - waits for operator

ENTER Key - Credential Validation

1. CICS resumes: EIBAID = DFHENTER, EIBCALEN > 0
2. EXEC CICS RECEIVE MAP('COSGN0A') MAPSET('COSGN00') - read screen input
3. Validate USERID field is not blank - if blank, error and redisplay
4. Validate PASSWD field is not blank - if blank, error and redisplay
5. Apply FUNCTION UPPER-CASE to USERID and PASSWD
6. EXEC CICS READ DATASET('USRSEC') INTO(SEC-USER-DATA) RIDFLD(WS-USER-ID)
7. Evaluate RESP: 0=found / 13=not found / other=error
8. If found: compare SEC-USR-PWD with WS-USER-PWD
9. If password matches: populate COMMAREA (USER-ID, USER-TYPE, FROM-TRANID, FROM-PROGRAM, PGM-CONTEXT=0)
10. Check SEC-USR-TYPE: 'A' -> XCTL COADM01C / else -> XCTL COMEN01C

PF3 - Exit

1. CICS resumes: EIBAID = DFHPF3
2. MOVE CCDA-MSG-THANK-YOU TO WS-MESSAGE
3. EXEC CICS SEND TEXT FROM(WS-MESSAGE) ERASE FREEKB
4. EXEC CICS RETURN (no TRANSID) - terminal session ends

Main Program Logic (from COSGN00C.cbl)

MAIN-PARA.
    SET ERR-FLG-OFF TO TRUE
    MOVE SPACES TO WS-MESSAGE
                   ERRMSGO OF COSGN0AO

    IF EIBCALEN = 0
        MOVE LOW-VALUES TO COSGN0AO
        MOVE -1         TO USERIDL OF COSGN0AI
        PERFORM SEND-SIGNON-SCREEN
    ELSE
        EVALUATE EIBAID
            WHEN DFHENTER
                PERFORM PROCESS-ENTER-KEY
            WHEN DFHPF3
                MOVE CCDA-MSG-THANK-YOU TO WS-MESSAGE
                PERFORM SEND-PLAIN-TEXT
            WHEN OTHER
                MOVE 'Y'                  TO WS-ERR-FLG
                MOVE CCDA-MSG-INVALID-KEY TO WS-MESSAGE
                PERFORM SEND-SIGNON-SCREEN
        END-EVALUATE
    END-IF.

    EXEC CICS RETURN
              TRANSID (WS-TRANID)
              COMMAREA (CARDDEMO-COMMAREA)
              LENGTH(LENGTH OF CARDDEMO-COMMAREA)
    END-EXEC.

VSAM Read with Full Error Handling (from COSGN00C.cbl)

READ-USER-SEC-FILE.
    EXEC CICS READ
         DATASET   (WS-USRSEC-FILE)       *> 'USRSEC  '
         INTO      (SEC-USER-DATA)         *> CSUSR01Y structure
         LENGTH    (LENGTH OF SEC-USER-DATA)
         RIDFLD    (WS-USER-ID)            *> 8-char uppercased key
         KEYLENGTH (LENGTH OF WS-USER-ID)
         RESP      (WS-RESP-CD)
         RESP2     (WS-REAS-CD)
    END-EXEC.

    EVALUATE WS-RESP-CD
        WHEN 0                            *> DFHRESP(NORMAL)
            IF SEC-USR-PWD = WS-USER-PWD
                MOVE WS-TRANID    TO CDEMO-FROM-TRANID
                MOVE WS-PGMNAME   TO CDEMO-FROM-PROGRAM
                MOVE WS-USER-ID   TO CDEMO-USER-ID

                        PROGRAM ('COADM01C')
                        COMMAREA(CARDDEMO-COMMAREA)
                    END-EXEC
                ELSE
                    EXEC CICS XCTL
                        PROGRAM ('COMEN01C')
                        COMMAREA(CARDDEMO-COMMAREA)
                    END-EXEC
                END-IF
            ELSE
                MOVE 'Wrong Password. Try again ...' TO WS-MESSAGE
                MOVE -1 TO PASSWDL OF COSGN0AI
                PERFORM SEND-SIGNON-SCREEN
            END-IF
        WHEN 13                           *> DFHRESP(NOTFND)
            MOVE 'Y' TO WS-ERR-FLG
            MOVE 'User not found. Try again ...' TO WS-MESSAGE
            MOVE -1  TO USERIDL OF COSGN0AI
            PERFORM SEND-SIGNON-SCREEN
        WHEN OTHER                        *> Unexpected error
            MOVE 'Y' TO WS-ERR-FLG
            MOVE 'Unable to verify the User ...' TO WS-MESSAGE
            MOVE -1  TO USERIDL OF COSGN0AI
            PERFORM SEND-SIGNON-SCREEN

PROCESS-ENTER-KEY.

              MAP('COSGN0A')
              MAPSET('COSGN00')

              RESP2(WS-REAS-CD)

12. Development Guidelines and Patterns

Pseudo-Conversational Pattern

CardDemo uses CICS pseudo-conversational programming. The program processes one screen interaction per invocation, then issues EXEC CICS RETURN TRANSID COMMAREA to suspend. When the operator presses a key, CICS re-invokes the program with the COMMAREA intact.

Key Rule: Never issue EXEC CICS RETURN without TRANSID except when intentionally ending the session (PF3 exit). Always pass COMMAREA on RETURN to preserve session state across interactions.

Error Flag Pattern

The module uses a two-value condition-name flag (WS-ERR-FLG) with 88-levels ERR-FLG-ON and ERR-FLG-OFF. Set with SET ERR-FLG-OFF TO TRUE at paragraph entry, and test with IF NOT ERR-FLG-ON before proceeding to VSAM read. This prevents VSAM reads when prior validation already failed.

COMMAREA Initialization

Authentication is the only program that creates a fresh COMMAREA. All other programs receive a pre-populated COMMAREA via XCTL or RETURN. When enhancing this module, always initialize all required COMMAREA fields to their defaults to prevent uninitialized data from propagating to downstream programs.

Cursor Positioning

Use MOVE -1 TO <fieldname>L OF <mapname>I to position the cursor at a specific field. This is the BMS length field mechanism for cursor positioning. The CURSOR option on EXEC CICS SEND activates this positioning.

13. User Stories and Acceptance Criteria

Story Templates

  • As a cardholder, I want to sign in with my user ID and password so I can access my account functions
  • As an administrator, I want to log in with admin credentials so I am routed to the admin menu
  • As a user, I want to receive a clear error when my credentials are wrong so I know to retry
  • As a user, I want to press PF3 to exit so I can safely end my session
  • As a system operator, I want the signon screen to show current date/time so I can verify system availability

Acceptance Criteria

Valid login - Regular User (BR-06, BR-10) Given a valid User ID and matching password for a 'U' type user in USRSEC, when the user presses ENTER, then the system XCTLs to COMEN01C (CM00), and CDEMO-USER-ID is set to the authenticated user, and CDEMO-USER-TYPE = 'U'.
Valid login - Administrator (BR-06, BR-09) Given a valid User ID and matching password for an 'A' type user in USRSEC, when the user presses ENTER, then the system XCTLs to COADM01C (CA00), and CDEMO-USER-TYPE = 'A' in CARDDEMO-COMMAREA.
Wrong password (BR-06, BR-12) Given a valid User ID but incorrect password, when user presses ENTER, then the signon screen redisplays with "Wrong Password. Try again ...", the cursor is on the Password field, and no user identity is set in COMMAREA.
User not found (BR-05) Given a User ID that does not exist in USRSEC, when user presses ENTER, then the signon screen redisplays with "User not found. Try again ...", and the cursor is on the User ID field.
Blank User ID (BR-01) Given the User ID field is empty, when user presses ENTER, then the screen redisplays with "Please enter User ID ..." and no VSAM read is attempted.
Blank Password (BR-02) Given a User ID is entered but Password is empty, when user presses ENTER, then the screen redisplays with "Please enter Password ..." and no VSAM read is attempted.
PF3 Exit (BR-07) Given any state on the signon screen, when user presses PF3, then the thank-you message is displayed via SEND TEXT and the CICS transaction ends (RETURN without TRANSID).
Invalid key (BR-08) Given any screen state, when user presses any key other than ENTER or PF3, then the signon screen redisplays with the standard invalid-key message and no credentials are processed.
Case-insensitive User ID (BR-03) Given user ID 'USERTEST' in USRSEC, when user enters 'usertest' (lowercase) with correct password, then authentication succeeds because FUNCTION UPPER-CASE is applied before the VSAM key lookup.

14. Risks and Tech Debt

HIGH - Plain-text passwords: SEC-USR-PWD stores passwords in plain text in the USRSEC VSAM file. Any user with read access to the VSAM dataset can view all system passwords. Modernization must implement cryptographic hashing (SHA-256 + salt) in a utility program before this is acceptable in any production environment.
MEDIUM - No account lockout: There is no mechanism to lock a user account after repeated failed login attempts. Brute-force credential guessing is unconstrained. Recommended: add a failed-attempt counter field to USRSEC (requires REWRITE access) and lockout after N failures.
MEDIUM - No session timeout: CICS pseudo-conversational session state is maintained indefinitely. An unattended authenticated terminal represents a security risk. Consider recording a session-start timestamp in COMMAREA and checking elapsed time on each transaction.
LOW - Hard-coded program names: Target programs 'COADM01C' and 'COMEN01C' are compile-time string literals. Any rename or routing change requires recompilation of COSGN00C and all programs that include COCOM01Y.
LOW - Read-only USRSEC access: COSGN00C only READs USRSEC; it never WRITEs or REWRITEs. Password change functionality requires a separate program with REWRITE access to the USRSEC file.

Recommended Modernization Order

  1. Authentication must work fully before any other module can be tested end-to-end
  2. Then: accounts, credit-cards, transactions, billing, reports, user-management, batch-processing
  3. Optional modules (authorization, transaction-types) only after core application is verified
