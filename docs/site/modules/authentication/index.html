🔐 Authentication

User Story Development Guide — CardDemo Mainframe Application

CICS COBOL VSAM Transaction: CC00

📋 Module Overview

Purpose: Handles user signon and session initialization. Validates credentials against the USRSEC VSAM file and routes users to the appropriate menu based on their user type (Admin or Regular).

Business Context: Entry point for all CardDemo functions. Every user interaction begins here. User type determined at login controls all subsequent navigation and access privileges.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • Cardholder login: As a cardholder, I want to sign in with my user ID and password so I can access my account functions
  • Admin login: As an administrator, I want to sign in with admin credentials so I am routed to the admin menu
  • Invalid login: As a cardholder, I want to receive a clear error message when my credentials are wrong so I know to retry
  • Session exit: As a user, I want to press PF3 to exit the application so I can safely end my session
  • Password policy: As an administrator, I want user passwords to be at least N characters so security standards are met

⚡ Development Acceleration Factors

  • COMMAREA pattern: Session state already structured in COCOM01Y — reuse for new screens without redesign
  • USRSEC KSDS: Direct keyed access by user ID — fast, no additional index needed for basic auth
  • PF key handling: Standard DFHAID / EIBAID pattern used across all CICS programs — consistent to implement
  • User type routing: CDEMO-USER-TYPE already propagated — conditional routing logic minimal

📊 Complexity Guidelines

Simple (1–2 pts) Update error message text; change screen field labels; add password masking attribute to BMS map
Medium (3–5 pts) Add account lockout after N failed attempts; add last-login timestamp to USRSEC record; implement session timeout
Complex (5–8 pts) Implement password hashing (requires utility program + USRSEC schema change); add RACF integration for external auth; support multi-factor authentication

🔧 Technical Foundation

Processing Flow

User types UserID + Password on COSGN00 screen
  → ENTER key triggers CC00 transaction
  → COSGN00C reads USRSEC VSAM (key = User ID)
  → Compares SEC-USR-PWD with entered password
  → On match: populates CARDDEMO-COMMAREA
      CDEMO-USER-ID = entered User ID
      CDEMO-USER-TYPE = 'A' (admin) or 'U' (user)
  → Routes to CM00 (Main Menu) or CA00 (Admin Menu)
  → On failure: redisplays signon screen with error

🔗 Dependencies

  • USRSEC VSAM file: Must be available — populated by DUSRSECJ JCL job before first use
  • DFHAID / DFHBMSCA: CICS system copybooks for key handling
  • COMMAREA: All downstream programs depend on CDEMO-USER-ID and CDEMO-USER-TYPE being set here

Data Model

SEC-USER-DATA (USRSEC VSAM, RECLN=80):
  SEC-USR-ID      PIC X(08)  ← Primary key
  SEC-USR-FNAME   PIC X(20)
  SEC-USR-LNAME   PIC X(20)
  SEC-USR-PWD     PIC X(08)  ← Plain text (demo only)
  SEC-USR-TYPE    PIC X(01)  ← 'A'=Admin, 'U'=User

✅ Acceptance Criteria Examples

Valid login — Regular User:
Given a valid User ID and matching password for a 'U' type user,
When user presses ENTER on the signon screen,
Then the system displays the Main Menu (CM00) and sets CDEMO-USER-TYPE = 'U'
Valid login — Admin:
Given a valid User ID and matching password for an 'A' type user,
When user presses ENTER,
Then the system displays the Admin Menu (CA00) and sets CDEMO-USER-TYPE = 'A'
Invalid credentials:
Given an incorrect password or non-existent User ID,
When user presses ENTER,
Then the signon screen is redisplayed with an error message and no session state is set
PF3 exit:
Given any state on the signon screen,
When user presses PF3,
Then a thank-you message is displayed and the session ends
Invalid key:
Given any screen state,
When user presses an unsupported PF key,
Then the screen redisplays with "Invalid Key" message and cursor repositioned on User ID field
