user-management

Administrative management of system users — list, add, update, and delete — for the CardDemo mainframe application.

CICS Online VSAM KSDS Admin Only JCL Init

Overview

The user-management module provides administrators with complete lifecycle management of user accounts stored in the USRSEC VSAM KSDS file. All operations—listing, adding, updating, and deleting users—are performed through CICS 3270 terminal screens and are exclusively available to users authenticated with Admin type (CDEMO-USER-TYPE = 'A').

This module is a core security boundary of the CardDemo system. The USRSEC file is shared with the authentication module: the signon program (COSGN00C) reads user credentials from this same file, while user-management programs write, update, and delete those records. Proper administration of the USRSEC file directly controls who can access the application and with what privilege level.

Admin-only access: Every program in this module checks the COMMAREA field CDEMO-USER-TYPE. Any non-admin session (or a session with no COMMAREA) is immediately redirected to the signon screen (COSGN00C).

Architectural Context

Position in the System

User Management sits at the top of the administrative branch of CardDemo's CICS program hierarchy. After signon (CC00 / COSGN00C), admin users are routed to the Admin Menu (CA00 / COADM01C), from which all user management functions are reached. Regular users never see the Admin Menu; they are routed to the Main Menu (CM00 / COMEN01C).

Technology Stack

  • Runtime: IBM CICS (transaction-per-screen model)
  • Language: COBOL (ANSI 85 / IBM Enterprise COBOL)
  • Screen Interface: BMS (Basic Mapping Support) 3270 terminal maps
  • Data Store: VSAM KSDS — USRSEC file (RECLN=80, KEY=8 bytes at offset 0)
  • Session State: COMMAREA (CARDDEMO-COMMAREA from COCOM01Y.cpy)
  • Initial Data Load: JCL job DUSRSECJ using IEBGENER + IDCAMS REPRO

Data Model

SEC-USER-DATA — USRSEC VSAM KSDS Record (80 bytes)

01 SEC-USER-DATA.
   05 SEC-USR-ID       PIC X(08).  *> Primary key — 8-char alphanumeric User ID
   05 SEC-USR-FNAME    PIC X(20).  *> First name (left-justified, space-padded)
   05 SEC-USR-LNAME    PIC X(20).  *> Last name
   05 SEC-USR-PWD      PIC X(08).  *> Password — plain text, max 8 chars (demo design)
   05 SEC-USR-TYPE     PIC X(01).  *> 'A' = Admin, 'U' = Regular User
   05 SEC-USR-FILLER   PIC X(23).  *> Unused filler; total record = 80 bytes

Processing Flows

CA00 — Admin Menu (COADM01C)

1
Receives COMMAREA; if EIBCALEN=0 (no session), XCTLs to COSGN00C immediately.
2
On first entry (CDEMO-PGM-CONTEXT=0): sends Admin Menu screen listing 6 options.
3
On ENTER: reads option number, validates it is numeric and within 1–CDEMO-ADMIN-OPT-COUNT (6).
4
Looks up CDEMO-ADMIN-OPT-PGMNAME(option) in the option table; checks for "DUMMY" placeholder (disabled option).
5
XCTLs to the target program with COMMAREA. A PGMIDERR HANDLE CONDITION traps missing programs gracefully.
6
On PF3: XCTLs to COSGN00C (signon screen).

CU00 — User List (COUSR00C)

1
On first entry, starts VSAM STARTBR from beginning of USRSEC file.
2
Reads up to 10 records with READNEXT; populates screen rows with User ID, Name (FNAME + LNAME), and User Type.
3
Saves first/last User ID of current page in COMMAREA (CDEMO-CU00-USRID-FIRST/LAST); sets NEXT-PAGE-FLG='Y' if more records exist.
4
PF7: repositions STARTBR to CDEMO-CU00-USRID-FIRST and reads backward (previous page). PF8: restarts from CDEMO-CU00-USRID-LAST + 1 (next page).
5
On ENTER with a row selection: if selection flag is 'U', stores selected User ID in CDEMO-CU00-USR-SELECTED and XCTLs to COUSR02C; if 'D', XCTLs to COUSR03C.
6
On PF3: returns to Admin Menu (COADM01C).

CU01 — Add User (COUSR01C)

1
On first entry: displays blank Add User form; positions cursor on FNAME field.
2
On ENTER: validates each required field in sequence (FNAME, LNAME, USERID, PASSWORD, USRTYPE). First blank field triggers error, sets cursor on offending field, redisplays.
3
Moves screen fields to SEC-USER-DATA working storage: USERIDI→SEC-USR-ID, FNAMEI→SEC-USR-FNAME, etc.
4
Executes CICS WRITE DATASET('USRSEC') ... RIDFLD(SEC-USR-ID).
5
On DFHRESP(NORMAL): clears all screen fields; shows green success message "User <id> has been added..."
6
On DFHRESP(DUPKEY/DUPREC): shows red error "User ID already exist..." and positions cursor on USERID field.

CU02 — Update User (COUSR02C)

1
Reads CDEMO-CU02-USR-SELECTED (target User ID passed from CU00 via COMMAREA).
2
Executes CICS READ DATASET('USRSEC') ... UPDATE to lock record; populates screen fields with current values.
3
On ENTER: validates all required fields are non-blank; if any field changed sets USR-MODIFIED-YES flag.
4
If modified: executes CICS REWRITE DATASET('USRSEC') with updated SEC-USER-DATA.
5
Displays green success message on DFHRESP(NORMAL); red error message on any failure.
6
On PF3: cancels without writing and returns to Admin Menu.

CU03 — Delete User (COUSR03C)

1
Reads target User ID from COMMAREA (CDEMO-CU02-USR-SELECTED — shared field with CU02).
2
Executes CICS READ ... UPDATE to lock record; displays user details (ID, name, type) for operator confirmation.
3
On ENTER (confirmation): performs READ UPDATE again, then executes CICS DELETE DATASET('USRSEC') RIDFLD(SEC-USR-ID).
4
On DFHRESP(NORMAL): displays green success "User <id> has been deleted..."
5
On PF3: cancels delete without making changes; returns to Admin Menu.

Business Rules

  • Admin-only access: CDEMO-USER-TYPE in COMMAREA must equal 'A'. If EIBCALEN = 0 (no CICS session established), any program immediately XCTLs to COSGN00C to force re-authentication.
  • User ID uniqueness: VSAM KSDS primary key constraint enforces uniqueness. COUSR01C explicitly handles DFHRESP(DUPKEY) and DFHRESP(DUPREC) with a user-facing error message rather than abending.
  • User ID format: 8-character left-justified alphanumeric string (e.g., ADMIN001, USER0001). All 8 bytes form the VSAM key; trailing spaces are part of the key value.
  • User type values: Only 'A' (Admin) and 'U' (Regular User) are meaningful. The user type stored in USRSEC is loaded into COMMAREA at signon and controls menu routing for the entire session.
  • All fields required on add: First Name, Last Name, User ID, Password, and User Type must all be non-blank. Field validation is sequential: the first blank field triggers an error. Each validation targets cursor placement to the offending field.
  • Password constraints: Maximum 8 characters, stored in plain text (demo environment characteristic). No complexity, history, or expiry requirements are enforced in the base application.
  • Hard delete: User deletion removes the VSAM record permanently using CICS DELETE. There is no soft-delete or deactivation flag; deleted users immediately lose the ability to log in.
  • Delete confirmation: COUSR03C is a two-step operation. The user must first view the record details, then press ENTER to confirm deletion. PF3 at any point cancels without making changes.
  • Pagination: CU00 displays exactly 10 users per page. CDEMO-CU00-NEXT-PAGE-FLG is set to 'Y' when additional records exist. Page state (first/last IDs, page number) is preserved in COMMAREA across pseudo-conversational returns.
  • Record locking: Both update (COUSR02C) and delete (COUSR03C) operations use CICS READ ... UPDATE to obtain an exclusive lock before presenting data for modification, preventing lost updates from concurrent sessions.
  • PGMIDERR handling: COADM01C registers a HANDLE CONDITION PGMIDERR so that optional programs (e.g., DB2-dependent menu options not installed) fail gracefully with a message rather than causing a transaction abend.
  • COMMAREA required: All programs test EIBCALEN = 0 at entry. A zero-length COMMAREA means no authenticated session exists; the program routes immediately to signon without attempting any VSAM I/O.

Initial Data Load

The JCL job DUSRSECJ initializes the USRSEC file in three ordered steps:

StepProgramAction
PREDELIEFBR14Pre-deletes the sequential PS dataset (AWS.M2.CARDDEMO.USRSEC.PS) if it already exists; prevents IEBGENER DISP=NEW failure
STEP01IEBGENERCreates the sequential PS file from in-stream data containing 10 fixed-length 80-byte user records (5 admin, 5 regular)
STEP02IDCAMSDeletes (if present) and redefines the VSAM KSDS cluster with KEYS(8,0), RECSIZE(80,80)
STEP03IDCAMS REPROCopies the sequential PS data into the VSAM KSDS file
Warning: Running DUSRSECJ resets the USRSEC file to the initial 10 users, permanently destroying any user records added through the CICS application. Use only for initial environment setup or controlled test data reset.

Field Validation Pattern on Add (COUSR01C)

PROCESS-ENTER-KEY.
    EVALUATE TRUE
        WHEN FNAMEI OF COUSR1AI = SPACES OR LOW-VALUES
            MOVE 'Y'  TO WS-ERR-FLG
            MOVE 'First Name can NOT be empty...' TO WS-MESSAGE
            MOVE -1   TO FNAMEL OF COUSR1AI
            PERFORM SEND-USRADD-SCREEN
        WHEN LNAMEI OF COUSR1AI = SPACES OR LOW-VALUES
            MOVE 'Y'  TO WS-ERR-FLG
            MOVE 'Last Name can NOT be empty...' TO WS-MESSAGE
            MOVE -1   TO LNAMEL OF COUSR1AI
            PERFORM SEND-USRADD-SCREEN
        WHEN USERIDI OF COUSR1AI = SPACES OR LOW-VALUES
            MOVE 'Y'  TO WS-ERR-FLG
            MOVE 'User ID can NOT be empty...' TO WS-MESSAGE
            MOVE -1   TO USERIDL OF COUSR1AI
            PERFORM SEND-USRADD-SCREEN
        WHEN PASSWDI OF COUSR1AI = SPACES OR LOW-VALUES
            MOVE 'Y'  TO WS-ERR-FLG
            MOVE 'Password can NOT be empty...' TO WS-MESSAGE
            MOVE -1   TO PASSWDL OF COUSR1AI
            PERFORM SEND-USRADD-SCREEN
        WHEN USRTYPEI OF COUSR1AI = SPACES OR LOW-VALUES
            MOVE 'Y'  TO WS-ERR-FLG
            MOVE 'User Type can NOT be empty...' TO WS-MESSAGE
            MOVE -1   TO USRTYPEL OF COUSR1AI
            PERFORM SEND-USRADD-SCREEN
        WHEN OTHER
            MOVE -1   TO FNAMEL OF COUSR1AI
            CONTINUE
    END-EVALUATE.

CICS WRITE — Create New User (COUSR01C)

WRITE-USER-SEC-FILE.
    EXEC CICS WRITE
         DATASET   (WS-USRSEC-FILE)
         FROM      (SEC-USER-DATA)
         LENGTH    (LENGTH OF SEC-USER-DATA)
         RIDFLD    (SEC-USR-ID)
         KEYLENGTH (LENGTH OF SEC-USR-ID)
         RESP      (WS-RESP-CD)
         RESP2     (WS-REAS-CD)
    END-EXEC

    EVALUATE WS-RESP-CD
        WHEN DFHRESP(NORMAL)

                   ' has been added ...' DELIMITED BY SIZE
              INTO WS-MESSAGE

            MOVE 'User ID already exist...' TO WS-MESSAGE
            MOVE -1 TO USERIDL OF COUSR1AI
            PERFORM SEND-USRADD-SCREEN
        WHEN OTHER
            MOVE 'Y' TO WS-ERR-FLG
            MOVE 'Unable to Add User...' TO WS-MESSAGE
            PERFORM SEND-USRADD-SCREEN
    END-EVALUATE.

READ-USER-SEC-FILE.
    EXEC CICS READ
         DATASET   (WS-USRSEC-FILE)

Paginated VSAM Browse (COUSR00C)

BROWSE-USER-SEC-FILE.
    EXEC CICS STARTBR
         DATASET   (WS-USRSEC-FILE)
         RIDFLD    (SEC-USR-ID)
         GTEQ
         RESP      (WS-RESP-CD)
    END-EXEC

    MOVE ZEROS TO WS-REC-COUNT
    PERFORM VARYING WS-IDX FROM 1 BY 1
            UNTIL WS-IDX > 10 OR USER-SEC-EOF
        EXEC CICS READNEXT
             DATASET   (WS-USRSEC-FILE)
             INTO      (SEC-USER-DATA)
             LENGTH    (LENGTH OF SEC-USER-DATA)
             RIDFLD    (SEC-USR-ID)
             RESP      (WS-RESP-CD)
        END-EXEC
        EVALUATE WS-RESP-CD
            WHEN DFHRESP(NORMAL)
                ADD 1 TO WS-REC-COUNT
                MOVE SEC-USR-ID    TO USER-ID(WS-IDX)
                MOVE SEC-USR-FNAME TO USER-NAME(WS-IDX)
                MOVE SEC-USR-TYPE  TO USER-TYPE(WS-IDX)
            WHEN DFHRESP(ENDFILE)
                SET USER-SEC-EOF TO TRUE
            WHEN OTHER
                SET USER-SEC-EOF TO TRUE
        END-EVALUATE
    END-PERFORM

    EXEC CICS ENDBR
         DATASET (WS-USRSEC-FILE)
    END-EXEC.

    *> Check if more pages exist after this one
    IF NOT USER-SEC-EOF
        SET NEXT-PAGE-YES TO TRUE
    END-IF.

Admin Menu Option Routing (COADM01C)

PROCESS-ENTER-KEY.
    *> Extract and right-justify numeric option from screen input
    PERFORM VARYING WS-IDX
            FROM LENGTH OF OPTIONI OF COADM1AI BY -1 UNTIL
            OPTIONI OF COADM1AI(WS-IDX:1) NOT = SPACES OR WS-IDX = 1
    END-PERFORM
    MOVE OPTIONI OF COADM1AI(1:WS-IDX) TO WS-OPTION-X
    INSPECT WS-OPTION-X REPLACING ALL ' ' BY '0'
    MOVE WS-OPTION-X TO WS-OPTION

    IF WS-OPTION IS NOT NUMERIC OR
       WS-OPTION > CDEMO-ADMIN-OPT-COUNT OR
       WS-OPTION = ZEROS
        MOVE 'Please enter a valid option number...' TO WS-MESSAGE
        PERFORM SEND-MENU-SCREEN
    END-IF

    IF NOT ERR-FLG-ON
        IF CDEMO-ADMIN-OPT-PGMNAME(WS-OPTION)(1:5) NOT = 'DUMMY'
            MOVE WS-TRANID  TO CDEMO-FROM-TRANID
            MOVE WS-PGMNAME TO CDEMO-FROM-PROGRAM
            MOVE ZEROS      TO CDEMO-PGM-CONTEXT
            EXEC CICS
                XCTL PROGRAM(CDEMO-ADMIN-OPT-PGMNAME(WS-OPTION))
                COMMAREA(CARDDEMO-COMMAREA)
            END-EXEC
        END-IF
    END-IF.

Development Guidelines

Adding a New Admin Menu Option

  1. Open app/cpy/COADM02Y.cpy
  2. Increment CDEMO-ADMIN-OPT-COUNT by 1
  3. Add a new 45-byte FILLER group: option number (2 bytes, PIC 9(02)), display text (35 bytes, PIC X(35)), program name (8 bytes, PIC X(08))
  4. Recompile COADM01C and any other programs that COPY COADM02Y
  5. Define the new target program in CICS CSD (DEFINE PROGRAM) and link the compiled load module
  6. Add a HANDLE CONDITION PGMIDERR in COADM01C if the new program may be absent in some environments

Validation and Error Handling Patterns

All user-management programs follow a consistent error-handling pattern:

  • Use WS-ERR-FLG (PIC X(01), 88-levels ERR-FLG-ON/OFF) to track error state within a transaction
  • Set cursor position using negative length field (e.g., MOVE -1 TO FNAMEL OF COUSR1AI) to position the 3270 cursor on the error field
  • Always evaluate CICS RESP codes using DFHRESP(condition) constants rather than numeric values
  • Display success messages in DFHGREEN; error messages naturally appear in the default color (red/neutral)

Pseudo-Conversational COMMAREA Pattern

All programs use the pseudo-conversational programming model: each CICS RETURN passes the COMMAREA back to the next invocation of the same transaction, preserving state across screen interactions.

*--- Return control to same transaction with state preserved ---
EXEC CICS RETURN
     TRANSID  (WS-TRANID)
     COMMAREA (CARDDEMO-COMMAREA)
END-EXEC.

*--- Transfer to another program (XCTL) ---
MOVE WS-TRANID  TO CDEMO-FROM-TRANID
MOVE WS-PGMNAME TO CDEMO-FROM-PROGRAM
MOVE ZEROS      TO CDEMO-PGM-CONTEXT
EXEC CICS XCTL
     PROGRAM (CDEMO-TO-PROGRAM)
     COMMAREA(CARDDEMO-COMMAREA)
END-EXEC.

Security Considerations for Modernization

Plain-text passwords: SEC-USR-PWD PIC X(08) is stored in plain text in USRSEC. Modernization must replace this with a hashed credential store (bcrypt or Argon2) and ensure passwords are never displayed on update screens.
No audit trail: Deletes are hard deletes with no logging. Add an audit log (DB2 table or z/OS SMF record) to capture who deleted which user ID and when.
No session invalidation: If an admin deletes a user who is currently logged in, that user's existing CICS session (with their COMMAREA) is not invalidated. Sessions expire only when the 3270 terminal disconnects or the CICS transaction times out.
