👥 User Management

User Story Development Guide — CardDemo Mainframe Application

CICS Admin Only COBOL VSAM USRSEC CA00 · CU00–CU03

📋 Module Overview

Purpose: Administrative management of system users — list, add, update, and delete user accounts. Controls who can access the CardDemo application and with what privileges.

Business Context: Admin-only module. User type ('A' or 'U') stored in USRSEC determines which menu and functions are accessible. User management is the access control mechanism for the entire application.

⚠️ Admin Access Required: All user management functions require CDEMO-USER-TYPE = 'A' in COMMAREA. Regular users cannot access CA00 or CU00–CU03.

🎯 User Story Development Guide

📝 Module-Specific US Templates

  • List users: As an administrator, I want to view all system users so I can audit who has access to CardDemo
  • Add user: As an administrator, I want to add a new user with ID, name, password, and type so they can access the system
  • Update user: As an administrator, I want to update a user's type from 'U' to 'A' so they have admin privileges
  • Delete user: As an administrator, I want to delete a user account so terminated employees lose system access
  • Password reset: As an administrator, I want to reset a user's password so locked-out users can regain access
  • User type control: As an administrator, I want to change a user's type so their access level matches their role

⚡ Development Acceleration Factors

  • USRSEC KSDS: Simple primary-key access by SEC-USR-ID — all CRUD operations are direct keyed reads/writes
  • Admin type check: CDEMO-USER-TYPE = 'A' check already implemented — reuse this pattern for any new admin function
  • Four-program CRUD: List/Add/Update/Delete already separated — each can be modified independently
  • DUSRSECJ JCL: Bulk user load pattern exists — reuse for user import from external systems

📊 Complexity Guidelines

Simple (1–2 pts) Add display-only field to user list; change field order on add/update screens; improve error message text
Medium (3–5 pts) Add user active/inactive flag to USRSEC; implement soft delete (deactivate vs. hard delete); add user search by name
Complex (5–8 pts) Add audit log for user changes (new VSAM file); implement password complexity rules (utility program); add user groups/roles beyond A/U types; bulk user import from external source

🔧 Technical Foundation

Add User Flow

Admin on CU01 screen:
  → Enter: User ID, First Name, Last Name, Password, Type
  → Validate: ID unique (READ USRSEC → NOTFOUND expected)
  → Validate: type = 'A' or 'U', all required fields present
  → WRITE USRSEC record (SEC-USER-DATA)
  → Display confirmation

Update User Flow

Admin selects user from list (CU00) → navigates to CU02:
  → READ USRSEC (key = selected SEC-USR-ID)
  → Display current values on screen
  → Admin modifies fields → ENTER
  → Validate changes
  → REWRITE USRSEC record
  → Display confirmation

Delete User Flow

Admin selects user from list → navigates to CU03:
  → Display user details for confirmation
  → Admin confirms deletion → ENTER
  → DELETE from USRSEC (key = SEC-USR-ID)
  → Hard delete — record permanently removed
  → Return to user list with confirmation

Data Model

SEC-USER-DATA (USRSEC VSAM, RECLN=80):
  SEC-USR-ID      PIC X(08)  ← Primary key (unique, 8 char)
  SEC-USR-FNAME   PIC X(20)  ← First name
  SEC-USR-LNAME   PIC X(20)  ← Last name
  SEC-USR-PWD     PIC X(08)  ← Password (plain text — demo only)
  SEC-USR-TYPE    PIC X(01)  ← 'A'=Admin, 'U'=User
  SEC-USR-FILLER  PIC X(23)  ← Reserved

🔗 Dependencies

  • authentication: COMMAREA must have CDEMO-USER-TYPE = 'A' — user management blocked for non-admins
  • USRSEC VSAM file: Must be pre-loaded via DUSRSECJ JCL before first admin login
  • RACF (optional): May integrate with external RACF security for enterprise deployments

✅ Acceptance Criteria Examples

List users — admin only:
Given a signed-in admin (CDEMO-USER-TYPE='A'),
When navigating to User List (CU00),
Then all users in USRSEC are displayed with ID, name, and type
Non-admin blocked:
Given a signed-in regular user (CDEMO-USER-TYPE='U'),
When attempting to navigate to CA00 or CU00,
Then access is denied with an "Unauthorized" message and user is returned to Main Menu
Add user — success:
Given all required fields entered on CU01 with a unique User ID,
When pressing ENTER,
Then a new SEC-USER-DATA record is written to USRSEC; success message displayed; user appears in list
Add user — duplicate ID:
Given a User ID that already exists in USRSEC,
When pressing ENTER on CU01,
Then "User ID already exists" error is displayed and no USRSEC write occurs
Delete user — confirmed:
Given an admin confirms deletion on CU03,
When pressing ENTER,
Then the record is permanently removed from USRSEC and user can no longer log in
Update user type:
Given an admin changes SEC-USR-TYPE from 'U' to 'A' on CU02,
When pressing ENTER,
Then USRSEC is updated; the user's next login routes them to the Admin Menu
